## 1. 뉴스 요약

2026년 4월, "바이브 코딩(Vibe Coding)" 열풍으로 앱스토어 신규 등록 건수가 84% 급증했다는 보도가 나왔다. 바이브 코딩은 자연어 프롬프트만으로 LLM이 코드를 작성·수정·배포까지 수행하는 새로운 개발 패러다임으로, 비전문가도 단시간에 앱을 출시할 수 있게 만들었다. 그러나 애플은 품질·보안 검수 부담 폭증을 이유로 등록 심사에 제동을 걸기 시작했다. 검증되지 않은 의존성, 환각(hallucination)에 의한 결함 코드, 라이선스 위반, 개인정보 처리 미흡 등이 주요 리스크로 지적되며, AI 생성 코드의 품질 거버넌스가 소프트웨어공학의 새로운 화두로 부상했다.

## 2. 핵심 개념

### 바이브 코딩 워크플로우와 위험 지점

```
┌─────────────────────────────────────────────────────────────┐
│              Vibe Coding 파이프라인과 리스크                  │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  자연어 프롬프트 ──→ LLM 코드 생성 ──→ 자동 빌드 ──→ 배포    │
│       │                  │                │           │     │
│       ▼                  ▼                ▼           ▼     │
│  요구 모호성       환각/오류 코드     의존성 자동      검수   │
│  보안 요건 누락    라이선스 미확인    설치(공급망)     생략   │
│                    민감정보 노출                              │
│                                                             │
│  ※ 전통 SDLC의 설계·리뷰·테스트 단계가 압축되거나 생략됨    │
└─────────────────────────────────────────────────────────────┘
```

### 전통 개발 vs 바이브 코딩 비교

| 구분 | 전통 SDLC | 바이브 코딩 |
|---|---|---|
| 요구 정의 | 명세서·UML | 자연어 프롬프트 |
| 설계 검토 | 아키텍처 리뷰 | LLM 자동 결정 |
| 코드 작성 | 개발자 수작업 | LLM 자동 생성 |
| 테스트 | 단위·통합·인수 | 생성 시 일부만 |
| 보안 검수 | SAST/DAST/SCA | 거의 부재 |
| 릴리스 주기 | 주·월 단위 | 분·시간 단위 |
| 책임 소재 | 개발 조직 | 모호(프롬프터?) |

### AI 코드 품질 거버넌스 프레임워크

```
[Prompt Layer]      프롬프트 표준화, 보안 가드레일
       ↓
[Generation Layer]  모델 선택, 사용 모델·버전 추적(SBOM-AI)
       ↓
[Validation Layer]  SAST·SCA·라이선스 스캔 자동 게이팅
       ↓
[Review Layer]      Human-in-the-Loop 코드 리뷰
       ↓
[Release Layer]     서명·검증, 앱스토어 정책 준수
       ↓
[Monitor Layer]     런타임 이상탐지, 취약점 피드백 루프
```

## 3. 기술사 출제 포인트

- **(논술)** 생성형 AI 기반 코드 작성(바이브 코딩) 확산이 소프트웨어 개발 생명주기(SDLC)에 미치는 영향을 분석하고, 품질·보안 거버넌스 확보 방안을 제시하시오.
- **(논술)** AI 생성 코드의 저작권·라이선스·책임 문제를 법·기술적 관점에서 논하고, 기업의 대응 전략을 제시하시오.
- **(약술)** SAST, DAST, SCA, IAST의 차이와 AI 코드 검증에 적합한 조합을 설명하시오.
- **(약술)** SBOM(Software Bill of Materials)과 AI 모델용 확장 개념(AI-BOM, MBOM)의 필요성을 서술하시오.
- **(약술)** Human-in-the-Loop(HITL) 코드 리뷰의 절차와 한계를 설명하시오.
- **출제 트렌드**: DevSecOps, Shift-Left Security, 공급망 보안(SLSA, SSDF), AI 시대 소프트웨어 품질 보증.

## 4. 관련 토픽 연계

- **DevSecOps**: CI/CD 파이프라인 내 보안 자동화, 정책 코드(Policy as Code)
- **공급망 보안**: SLSA(Supply-chain Levels for Software Artifacts), NIST SSDF, 의존성 변조 방지
- **소프트웨어 품질**: ISO/IEC 25010 품질 모델(기능적합성·신뢰성·보안성·유지보수성)
- **AI 거버넌스**: ISO/IEC 42001(AI Management System), EU AI Act의 고위험 AI 요구사항
- **앱스토어 정책**: Apple App Review Guidelines, Google Play Policy, 사이드로딩 규제(EU DMA)
- **저작권**: AI 학습 데이터 저작권, 생성 결과물의 저작권 귀속, 오픈소스 라이선스 준수
- **개발 생산성**: DORA 4지표(Deployment Frequency, Lead Time, MTTR, Change Failure Rate)
- **관련 토픽**: 에이전틱 AI, LLM 추론 비용 최적화, AI 코드 어시스턴트(GitHub Copilot, Cursor)
