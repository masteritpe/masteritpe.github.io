## 1. 뉴스 요약

2026년 4월, 양자컴퓨터가 비트코인 암호를 9분 만에 해독할 수 있다는 연구 결과가 발표되면서 "Harvest Now, Decrypt Later(HNDL)" 공격에 대한 우려가 급증하고 있다. HNDL은 현재 암호화된 데이터를 대량 수집한 뒤, 양자컴퓨터가 실용화되는 시점에 일괄 복호화하는 공격 전략이다. 특히 "소급 암호 해독의 함정"이라는 분석에서는 양자 시대 이후 암호를 업그레이드하더라도 과거에 수집된 데이터는 보호할 수 없다는 점을 경고했다. 알고랜드(Algorand) 등 블록체인 프로젝트는 양자내성암호(PQC)를 선제 적용하며 대응에 나서고 있으며, 이는 모든 산업에서 암호 민첩성(Crypto Agility) 확보가 시급함을 보여준다.

## 2. 핵심 개념

### HNDL(Harvest Now, Decrypt Later) 공격 메커니즘

```
┌─────────────────────────────────────────────────────────┐
│              HNDL 공격 타임라인                           │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  현재(2026)              양자 실용화(202X)               │
│  ┌──────────┐           ┌──────────────────┐           │
│  │ 공격자가  │           │ 양자컴퓨터로     │           │
│  │ 암호화된  │ ────────→ │ 과거 수집 데이터 │           │
│  │ 트래픽    │  데이터   │ 일괄 복호화      │           │
│  │ 대량 수집 │  보관     │                  │           │
│  └──────────┘           └──────────────────┘           │
│       ↑                        ↓                        │
│  RSA/ECC 기반            기밀정보·개인정보               │
│  현행 암호 사용          대규모 유출                     │
│                                                         │
│  ※ 핵심: 암호 업그레이드 후에도 과거 데이터는 보호 불가  │
└─────────────────────────────────────────────────────────┘
```

### HNDL 위협 대상과 영향도

| 위협 대상 | 데이터 유효 기간 | HNDL 위험도 | 사례 |
|----------|----------------|------------|------|
| 국가 기밀·외교 문서 | 30~50년 | 매우 높음 | 정부 간 암호화 통신 |
| 의료·유전체 데이터 | 평생(50년+) | 매우 높음 | 전자건강기록(EHR) |
| 금융 거래·계좌 정보 | 10~20년 | 높음 | TLS 기반 온라인 뱅킹 |
| 기업 영업비밀 | 5~15년 | 높음 | VPN 통신, 클라우드 스토리지 |
| 블록체인·암호화폐 | 영구 | 매우 높음 | 비트코인 ECDSA 서명 |
| 일반 개인정보 | 3~7년 | 중간 | HTTPS 웹 트래픽 |

### 암호 민첩성(Crypto Agility) 아키텍처

```
┌─────────────────────────────────────────────────────────┐
│           암호 민첩성(Crypto Agility) 계층 구조          │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  ┌─────────────────────────────────────────────────┐   │
│  │         응용 계층 (Application Layer)            │   │
│  │  • 암호 알고리즘 추상화 인터페이스               │   │
│  │  • 알고리즘 선택을 설정(Config)으로 분리         │   │
│  └──────────────────────┬──────────────────────────┘   │
│                          ↓                              │
│  ┌─────────────────────────────────────────────────┐   │
│  │       암호 추상화 계층 (Crypto Abstraction)      │   │
│  │  ┌──────────┐  ┌──────────┐  ┌──────────────┐  │   │
│  │  │ 현행 암호 │  │하이브리드│  │ PQC 전용     │  │   │
│  │  │ RSA/ECC  │  │현행+PQC  │  │ ML-KEM 등   │  │   │
│  │  └──────────┘  └──────────┘  └──────────────┘  │   │
│  │         ↕ 런타임 알고리즘 전환(Hot Swap)         │   │
│  └──────────────────────┬──────────────────────────┘   │
│                          ↓                              │
│  ┌─────────────────────────────────────────────────┐   │
│  │       인프라 계층 (Infrastructure Layer)          │   │
│  │  • HSM/KMS 양자내성 키 관리                      │   │
│  │  • 인증서 이중 발급(현행 + PQC)                  │   │
│  │  • 프로토콜 협상(TLS 1.3 + PQ 확장)             │   │
│  └─────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────┘
```

### 현행 암호 vs 포스트양자암호(PQC) 비교

| 구분 | 현행 암호 | 포스트양자암호(PQC) |
|------|----------|-------------------|
| 키 교환 | RSA, ECDH | ML-KEM(CRYSTALS-Kyber) |
| 전자서명 | RSA, ECDSA | ML-DSA(CRYSTALS-Dilithium), SLH-DSA(SPHINCS+) |
| 양자 내성 | 없음(Shor 알고리즘에 취약) | 격자(Lattice), 해시 기반으로 양자 내성 |
| 키 크기 | 작음(256bit ECC) | 큼(ML-KEM-768: 1,184byte 공개키) |
| 성능 | 빠름 | 키 교환은 유사, 서명 검증은 다소 느림 |
| 표준화 | NIST FIPS 186-5 등 | NIST FIPS 203, 204, 205 (2024 확정) |
| 전환 전략 | - | 하이브리드 모드 → 단독 PQC |

### 암호 마이그레이션 로드맵

```
Phase 1: 탐색·인벤토리 (현재)
├── 암호 자산 목록화(CBOM: Cryptographic Bill of Materials)
├── HNDL 위험도 평가(데이터 유효 기간 × 기밀 수준)
└── 양자 위협 타임라인 분석

Phase 2: 하이브리드 전환 (2025~2027)
├── TLS 1.3 + PQ 하이브리드 키 교환 적용
├── 이중 인증서(현행 + PQC) 발급 체계 구축
└── HSM/KMS PQC 알고리즘 지원 업그레이드

Phase 3: PQC 전면 전환 (2028~2030)
├── 현행 암호 의존성 완전 제거
├── PQC 전용 모드 운영
└── 암호 민첩성 상시 모니터링 체계 운영
```

## 3. 기술사 출제 포인트

### 예상 문제
- **HNDL(Harvest Now, Decrypt Later) 공격의 개념과 위협을 설명하고, 이에 대응하기 위한 암호 민첩성(Crypto Agility) 확보 전략을 논하시오**
- **포스트양자암호(PQC) 전환 시 하이브리드 암호 방식의 필요성과 구현 방안을 설명하시오**
- **CBOM(Cryptographic Bill of Materials)의 개념과 양자 시대 암호 자산 관리에서의 역할을 논하시오**

### 답안 작성 키워드
- HNDL: 현재 수집 → 미래 복호화, 데이터 유효 기간이 핵심 변수
- 암호 민첩성(Crypto Agility): 암호 알고리즘의 런타임 교체 가능 설계
- CBOM: 시스템 내 암호 사용 현황 목록화(어디서, 어떤 알고리즘, 어떤 키 길이)
- 하이브리드 암호: 현행(RSA/ECC) + PQC 이중 적용으로 점진적 전환
- NIST PQC 표준: ML-KEM(FIPS 203), ML-DSA(FIPS 204), SLH-DSA(FIPS 205)
- Q-Day: 양자컴퓨터가 현행 암호를 실시간 해독 가능한 시점

### HNDL 대응 체크리스트

| 단계 | 점검 항목 | 세부 내용 |
|------|----------|----------|
| 식별 | 암호 자산 인벤토리 | CBOM 작성, 사용 중인 알고리즘·키·인증서 목록화 |
| 평가 | HNDL 위험도 분류 | 데이터 유효 기간 × 기밀 수준 매트릭스 |
| 설계 | 암호 추상화 계층 도입 | 알고리즘 교체가 코드 변경 없이 가능한 구조 |
| 전환 | 하이브리드 모드 적용 | TLS, VPN, 스토리지 암호화에 PQC 병행 적용 |
| 검증 | 상호운용성 테스트 | PQC 적용 후 성능·호환성 검증 |
| 운영 | 지속 모니터링 | 양자 위협 동향 추적, 알고리즘 갱신 자동화 |

## 4. 관련 토픽 연계

| 연계 토픽 | 연관성 |
|----------|--------|
| 포스트양자암호(PQC) | HNDL 대응의 핵심 기술, NIST 표준화 완료 |
| 공개키 기반 구조(PKI) | PQC 전환 시 인증서·CA 체계 재설계 필요 |
| TLS 1.3 | PQ 하이브리드 키 교환 확장이 주요 전환 경로 |
| 제로트러스트 보안 | 암호 민첩성은 제로트러스트의 데이터 보호 계층 핵심 |
| 블록체인 보안 | ECDSA 서명 양자 취약성, PQC 서명 전환 필요 |
| 정보보호 관리체계(ISMS) | 암호 자산 관리·위험 평가에 HNDL 시나리오 반영 |
| 클라우드 보안 | CSP의 PQC 지원 현황과 고객 데이터 보호 전략 |
