## 1. 뉴스 요약

전자신문·연합뉴스·뉴스1·문화일보·더팩트·매일일보·뉴시스 등 다수 매체(2026-05-11)는 **서울시**가 **자체 LLM 구축을 완료**하고 **생성형 AI 기반 '챗봇 2.0'을 본격 가동**한다고 일제히 보도했다. 챗봇 2.0은 ① **공무원 업무지원(문서 작성·법령 검색·민원 응답 초안)**, ② **시민 상담(서울시 행정·복지·교통 정보 안내)**, ③ **AI 에이전트(다단계 업무 자동화: 예약·민원접수·통계 조회)**의 세 축으로 운영된다. 핵심은 ChatGPT 등 외부 API에 의존하지 않고 **서울시 자체 LLM**을 구축해 **개인정보·내부 행정 데이터 주권**을 유지하면서, **서울시 법규·조례·민원 데이터를 RAG로 결합**한다는 점이다. AI데이터분석 관점에서 이 사례는 **엔터프라이즈 LLM·RAG·AI 에이전트·MLOps·LLMOps**를 결합한 **도시 행정 생성형 AI 시스템 아키텍처**의 대표 레퍼런스가 된다.

## 2. 핵심 개념

### 자체 LLM 기반 행정 챗봇 2.0 정의

**자체 LLM 기반 행정 챗봇**은 외부 상용 API(OpenAI 등) 대신 **공공기관이 자체 구축·파인튜닝한 LLM**과 **내부 지식베이스(법령·조례·민원·매뉴얼)를 RAG로 결합**한 생성형 AI 시스템이다. **데이터 주권·개인정보 통제·맞춤형 도메인 응답**이 가능하다는 점에서 1세대 룰기반 챗봇과 본질적으로 다르다.

### 서울시 챗봇 2.0 시스템 아키텍처

```
┌──────────────────────────────────────────────────────────────┐
│ ① 사용자 채널: 공무원 포털·시민 웹/앱·민원24·콜센터          │
├──────────────────────────────────────────────────────────────┤
│ ② AI 에이전트 계층: 의도분류·툴호출·다단계 작업 오케스트레이션│
├──────────────────────────────────────────────────────────────┤
│ ③ RAG 파이프라인: 검색(BM25+Vector)·재순위(Rerank)·증거첨부 │
├──────────────────────────────────────────────────────────────┤
│ ④ 자체 LLM: 도메인 SFT/PEFT(LoRA)·서울시 코퍼스 튜닝         │
├──────────────────────────────────────────────────────────────┤
│ ⑤ 지식베이스: 법령·조례·매뉴얼·민원·통계(벡터DB+그래프DB)    │
├──────────────────────────────────────────────────────────────┤
│ ⑥ 툴 게이트웨이: 행정시스템 API(예약·통계·민원접수) 호출     │
├──────────────────────────────────────────────────────────────┤
│ ⑦ 가드레일: 프롬프트 인젝션 방어·개인정보 마스킹·환각 검출   │
├──────────────────────────────────────────────────────────────┤
│ ⑧ LLMOps: 모델 버전·평가·옵저버빌리티·피드백 학습루프        │
└──────────────────────────────────────────────────────────────┘
```

### 1세대 룰기반 챗봇 vs 자체 LLM 챗봇 2.0 비교

다음 표는 행정 챗봇의 세대별 차이를 정리한다.

| 구분 | 1세대 룰기반/시나리오 챗봇 | 2세대 자체 LLM 챗봇 |
|------|----------------------------|----------------------|
| 응답 방식 | 정해진 시나리오·키워드 매칭 | 자연어 생성·맥락 이해 |
| 지식 갱신 | 시나리오 수기 추가 | 문서 추가 → RAG 자동 반영 |
| 도메인 적응 | 시나리오별 별도 구축 | 파인튜닝(LoRA·SFT)으로 적응 |
| 다단계 업무 | 어려움(분기 한계) | AI 에이전트로 툴 연쇄 호출 |
| 응답 정확도 | 정형 질문에 한정 | 비정형 질문 처리 가능 |
| 데이터 주권 | 자체 구축이면 보장 | 자체 LLM이면 보장 |
| 환각 위험 | 거의 없음 | 존재(RAG·가드레일로 완화) |
| 운영 비용 | 시나리오 유지비 | GPU·LLMOps 비용 |
| 적합 업무 | 단순 FAQ | 문서 작성·민원 응답·업무 자동화 |

### 핵심 구성요소

- **자체 LLM(파운데이션 모델)**: 국산/오픈소스(Llama·Qwen·Gemma·HyperCLOVA 등) 기반, 서울시 도메인 SFT·DPO·LoRA 파인튜닝
- **RAG 파이프라인**: 임베딩 모델 → 벡터DB(pgvector·Milvus·Weaviate) → BM25 하이브리드 검색 → Cross-Encoder 리랭킹 → 컨텍스트 주입
- **지식베이스**: 법령·조례(국가법령정보), 행정 매뉴얼, 민원 데이터, 통계, 공지사항 — 정기 인덱싱·증분 업데이트
- **AI 에이전트**: ReAct·ToolUse 패턴, 행정 API(예약·통계·민원접수) 함수 호출, 다단계 작업 계획·실행
- **가드레일**: 프롬프트 인젝션 탐지(LLM Guard), 개인정보 마스킹(PII Detection), 환각 검출(RAGAS·Faithfulness), 금지어 필터
- **LLMOps 플랫폼**: 모델 레지스트리(MLflow), 평가(eval suite·LLM-as-Judge), 옵저버빌리티(LangSmith·Arize·Phoenix), 휴먼 피드백 루프
- **접근통제·감사**: 공무원 인증(SSO), 시민 본인확인, 모든 질의·응답 감사 로그, 개인정보 처리방침 준수
- **인프라**: GPU 클러스터(자체 또는 공공 AIDC), 추론 서빙(vLLM·TGI·TensorRT-LLM), 캐싱·배치

### 도입 시 고려사항

- **데이터 주권·개인정보**: 외부 API 의존 회피, 학습 데이터 익명화·비식별, 개인정보보호법 준수
- **환각·신뢰성**: RAG 증거 첨부, 출처 표시 의무화, 법률·정책 답변에 휴먼 검증 게이트
- **모델 선택**: 자체 학습 vs 오픈소스 파인튜닝 vs 국산 파운데이션 활용 — 비용·성능·주권 균형
- **RAG 품질**: 문서 청킹 전략, 임베딩 모델 한국어 성능, 하이브리드 검색·리랭킹 적용
- **AI 에이전트 안전성**: 툴 권한 최소화, 행위 승인 게이트, 비가역 작업(결제·접수)에 휴먼 확인
- **편향·공정성**: 행정 결정 지원 시 차별·편향 평가(공공 AI 윤리 가이드라인)
- **시민 접근성**: 다국어·고령자·장애인 접근성, 음성·텍스트 멀티모달
- **운영 비용**: GPU 자원 사용 최적화(양자화·KV 캐시·배치), 자체 vs 공공 AIDC 활용
- **개인정보 보호**: 민원 데이터 학습 시 가명화, 차분 프라이버시, 사용자 옵트아웃
- **거버넌스**: AI 도입 영향평가, 알고리즘 책임성, 정기 외부 감사

## 3. 기술사 출제 포인트

- 자체 LLM 기반 엔터프라이즈 챗봇의 개념 정의와 도입 필요성
- 1세대 룰기반 챗봇과 2세대 LLM 챗봇 아키텍처 비교
- RAG(Retrieval-Augmented Generation) 파이프라인 구성요소
- 하이브리드 검색(BM25 + Vector)과 리랭킹 전략
- 파인튜닝 기법(SFT·DPO·LoRA·QLoRA) 비교와 도메인 적응 전략
- AI 에이전트 아키텍처(ReAct·ToolUse·Function Calling)
- LLM 가드레일과 프롬프트 인젝션·환각 대응 방안
- LLMOps와 모델 평가·옵저버빌리티·피드백 학습 루프
- 공공 LLM의 데이터 주권·개인정보 보호 거버넌스
- 벡터DB(pgvector·Milvus·Weaviate) 선택 기준과 운영 전략
- 추론 서빙 최적화(vLLM·양자화·KV 캐시·배치)
- 행정 AI의 환각 위험과 휴먼 인 더 루프(HITL) 설계
