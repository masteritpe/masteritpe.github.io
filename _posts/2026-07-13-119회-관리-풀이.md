### 1교시형 문제 풀이

### 1. 제약이론 TOC (Theory of Constraints)  

**1) 제약이론(TOC)의 정의**  

* 시스템의 목표 달성을 저해하는 가장 큰 제약조건(Bottleneck)을 식별하고, 이를 집중적으로 개선하여 전체 시스템의 처리율(Throughput)을 최적화하는 경영 혁신 기법입니다.  

**2) TOC의 5단계 집중 개선 프로세스 (POOGI)**  

| 단계 | 설명 | 활동 내용 |
| --- | --- | --- |
| **1. 식별 (Identify)** | 제약조건 찾기 | 전체 공정 중 가장 속도가 느린 병목 구간 파악 |
| **2. 활용 (Exploit)** | 제약조건의 최대 활용 | 추가 투자 없이 병목 구간의 효율성을 극대화 |
| **3. 종속 (Subordinate)** | 비제약조건의 속도 조절 | 제약조건의 처리 속도에 맞춰 타 공정의 페이스 조절 |
| **4. 향상 (Elevate)** | 제약조건의 능력 향상 | 인력 충원, 설비 투자 등을 통해 병목 구간의 절대적 능력 증대 |
| **5. 반복 (Repeat)** | 타성 방지 및 피드백 | 제약조건이 해소되면 새로운 제약을 찾아 1단계로 회귀 |

**3) 기대효과 및 적용 분야**  

* 재고 감소, 리드타임 단축, 현금 흐름 개선에 기여하며, 소프트웨어 개발의 애자일/DevOps 파이프라인 병목 개선(예: CI/CD 최적화)에도 응용됩니다.

### 2. 가상화기술의 도커 (DOCKER)  

**1) 도커(Docker)의 정의**  

* 하이퍼바이저(Hypervisor) 없이 호스트 OS의 커널을 공유하면서 프로세스를 격리된 환경에서 실행하는 리눅스 컨테이너(LXC) 기반의 오픈소스 경량 가상화 플랫폼입니다.  

**2) 도커의 핵심 요소 기술 및 아키텍처**  

* **격리 기술 (Namespace):** 프로세스, 네트워크, 마운트 등의 자원을 독립된 공간으로 격리 (PID, NET, IPC 등).  
* **자원 제어 (cgroups):** 컨테이너별로 CPU, 메모리, I/O 등의 물리적 자원 사용량 제한.  
* **파일 시스템 (UFS):** Union File System을 통해 도커 이미지를 읽기 전용 레이어(Layer)로 구성하여 저장 공간 최소화 및 빠른 배포 지원.  

**3) 기존 VM(가상머신)과의 비교**  

| 구분 | 가상머신 (VM) | 도커 컨테이너 (Docker) |
| --- | --- | --- |
| **구조** | Host OS → Hypervisor → Guest OS | Host OS → Docker Engine → Container |
| **성능/크기** | 무거움 (GB 단위), 부팅 느림 | 가벼움 (MB 단위), 부팅 빠름 (초 단위) |
| **목적** | 완벽한 OS 환경 격리 | 애플리케이션과 종속성의 패키징 및 배포 |

### 3. DID (Decentralized Identity)  

**1) DID (분산 신원 증명)의 정의**  

* 중앙 집중형 신원 관리 기관(Identity Provider)의 통제 없이, 사용자 스스로 자신의 신원 정보를 생성하고 통제할 수 있는 자기주권신원(SSI, Self-Sovereign Identity) 기반의 인증 체계입니다.  

**2) DID의 핵심 구성 요소**  

| 구성요소 | 역할 및 특징 |
| --- | --- |
| **Issuer (발행자)** | 사용자의 신원을 확인하고 신원정보(VC, Verifiable Credential)를 발행하는 기관. |
| **Holder (소유자)** | 발행받은 VC를 자신의 전자지갑(Wallet)에 저장하고, 필요시 VP를 생성하여 제출하는 사용자. |
| **Verifier (검증자)** | Holder가 제출한 VP(Verifiable Presentation)를 블록체인의 DID Document를 통해 위변조 여부 검증. |
| **Data Registry** | DID와 DID Document가 저장되는 분산 원장(Blockchain). |

**3) 기술적 시사점**  

* 개인정보 유출 피해를 최소화하고 데이터 주권을 개인에게 환원하며, ZKP(영지식 증명)와 결합하여 최소한의 정보만 제공하는 프라이버시 보호 인증으로 진화하고 있습니다.

### 4. UX 기법인 Design Thinking  

**1) 디자인 씽킹(Design Thinking)의 정의**  

* 디자이너의 감수성과 작업 방식을 활용하여, 사용자의 숨겨진 니즈를 깊이 공감하고 비즈니스 문제를 창의적으로 해결하는 인간 중심(Human-centric)의 혁신 방법론입니다.  

**2) 디자인 씽킹의 5단계 프로세스 (스탠포드 d.school 모델)**  

1\. **공감 (Empathize):** 인터뷰, 섀도잉(Shadowing) 등을 통해 사용자의 관점과 감정을 깊이 이해.  
2\. **문제 정의 (Define):** 수집된 데이터를 바탕으로 사용자의 진짜 문제(Point of View)를 통찰하고 정의.  
3\. **아이디어 도출 (Ideate):** 브레인스토밍 등을 통해 제약 없이 다양하고 창의적인 해결책 발산.  
4\. **프로토타입 (Prototype):** 아이디어를 시각적이고 구체적인 형태(Low-fi 또는 Hi-fi)로 빠르게 제작.  
5\. **테스트 (Test):** 프로토타입을 실제 사용자에게 시연하고 피드백을 받아 개선안 도출(반복).  

**3) 활용 방안**  

* 기존의 폭포수형 기획 방식에서 벗어나, 애자일(Agile) 및 린 스타트업(Lean Startup)과 결합하여 신규 서비스 UX 기획 및 프로덕트 마켓 핏(PMF) 검증에 활용됩니다.

### 5. 데이터베이스 샤드 (Database Shard)  

**1) 데이터베이스 샤딩(Sharding)의 정의**  

* 단일 데이터베이스에 저장하기 벅찬 대용량 데이터를 논리적으로 분할하여, 물리적으로 다른 여러 대의 데이터베이스 서버(Node)에 분산 저장하고 처리하는 수평 파티셔닝(Horizontal Partitioning) 기술입니다.  

**2) 샤딩의 주요 기법**  

| 기법 | 분산 방식 | 특징 및 장단점 |
| --- | --- | --- |
| **Hash Sharding** | Shard Key의 해시값에 따라 데이터 분배 | 데이터 분산이 균일하나, 노드 증설 시 해시 재계산(Rehashing) 문제 발생 (Consistent Hashing으로 완화). |
| **Range Sharding** | Shard Key의 값의 범위(Range)를 기준으로 분배 | 쿼리 처리가 직관적이나, 특정 대역(예: 최신 데이터)에 트래픽이 몰리는 Hotspot 문제 발생 가능. |
| **Directory Sharding** | 별도의 매핑 테이블(Lookup Table)을 두어 분산 | 샤딩 로직이 유연하지만, 라우팅 테이블이 단일 장애점(SPOF)이 될 수 있음. |

**3) 도입 시 고려사항**  

* Join 연산의 복잡도 증가와 트랜잭션 관리(Global Transaction)의 어려움이 발생하므로, NoSQL 또는 분산 RDBMS 아키텍처 도입 시 데이터 접근 패턴을 면밀히 분석해야 합니다.

### 6. 기계학습의 Gradient descent algorithm  

**1) 경사하강법(Gradient Descent Algorithm)의 정의**  

* 기계학습 및 딥러닝에서 모델의 예측값과 실제값의 오차를 나타내는 비용 함수(Cost Function)의 미분값(Gradient)을 계산하여, 오차가 최소화되는 최적의 가중치(Weight)와 편향(Bias)을 반복적으로 찾아가는 최적화 알고리즘입니다.  

**2) 동작 원리 및 수식**  

* 현재 위치에서 비용 함수의 기울기(Gradient)를 구하고, 기울기의 반대 방향으로 학습률(Learning Rate)만큼 이동하며 가중치를 업데이트합니다.  
* **업데이트 수식:**  

$$\theta&#95;{j} := \theta&#95;{j} - \alpha \frac{\partial}{\partial \theta&#95;{j}} J(\theta)$$  



*(여기서 $\theta$는 가중치 파라미터, $\alpha$는 학습률(Learning Rate), $J(\theta)$는 비용 함수)*  

**3) 주요 발전 유형**  

* **BGD (Batch Gradient Descent):** 전체 데이터를 한 번에 학습하여 업데이트 (정확하지만 느림).  
* **SGD (Stochastic Gradient Descent):** 데이터 1개마다 가중치 업데이트 (빠르지만 불안정함).  
* **Mini-Batch SGD:** 전체 데이터를 N개의 배치로 나누어 학습 (BGD와 SGD의 절충안, 가장 널리 쓰임).

### 7. Lehman의 SW 변화원리  

**1) 리먼의 법칙 (Lehman's Laws of Software Evolution) 정의**  

* 소프트웨어는 개발 후 운영 환경에 배포된 이후에도 지속적으로 변경되고 진화하며, 이 과정에서 발생하는 시스템의 복잡도 증가와 유지보수의 한계를 설명한 8가지 진화 법칙입니다.  

**2) 주요 원리 (핵심 4법칙 중심)**  

| 법칙 | 명칭 | 주요 내용 |
| --- | --- | --- |
| **제1법칙** | 계속적 변경 (Continuing Change) | 환경에 적응하기 위해 소프트웨어는 끊임없이 변경되어야 하며, 그렇지 않으면 도태됨. |
| **제2법칙** | 복잡도 증가 (Increasing Complexity) | 변경이 누적될수록 구조적 복잡도는 증가하며, 이를 방지하기 위한 리팩토링 노력이 필요함. |
| **제3법칙** | 프로그램 진화 (Continuous Growth) | 시스템의 기능과 크기는 사용자 요구사항에 따라 지속적으로 증가함. |
| **제4법칙** | 친숙성 유지 (Declining Quality) | 운영 환경이 변함에 따라 소프트웨어의 품질은 상대적으로 저하되는 것으로 인식됨. |

**3) 시사점**  

* 소프트웨어 유지보수는 단순한 버그 수정이 아닌 지속적인 진화 과정이므로, 아키텍처 개선(Refactoring)과 기술 부채(Technical Debt) 관리가 필수적임을 시사합니다.

### 8. 데이터베이스의 Isolation Level  

**1) 트랜잭션 격리 수준(Isolation Level)의 정의**  

* 다중 프로그래밍 환경에서 여러 트랜잭션이 동시에 실행될 때, 특정 트랜잭션이 다른 트랜잭션이 변경하거나 조회하는 데이터에 접근할 수 있도록 허용하는 일관성 보장 수준입니다. (ACID 중 Isolation 특성)  

**2) ANSI SQL-92 기준 4단계 격리 수준 및 발생 가능한 부작용**  

| 레벨 | 명칭 | 특징 | 방지하지 못하는 부작용 |
| --- | --- | --- | --- |
| **Level 0** | **Read Uncommitted** | 커밋되지 않은 데이터도 읽기 허용 | Dirty Read, Non-Repeatable Read, Phantom Read |
| **Level 1** | **Read Committed** | 커밋이 완료된 데이터만 읽기 허용 | Non-Repeatable Read, Phantom Read |
| **Level 2** | **Repeatable Read** | 트랜잭션 종료 시까지 동일 결과 보장 | Phantom Read |
| **Level 3** | **Serializable** | 트랜잭션을 순차적으로 실행 (완벽한 격리) | 없음 (가장 안전하나 성능이 최하위) |

* *참고: 부작용(Side Effect) 정의*  
* **Dirty Read:** 아직 커밋되지 않은 데이터를 읽어 발생하는 논리적 오류.  
* **Non-Repeatable Read:** 한 트랜잭션 내에서 같은 조건으로 데이터를 두 번 읽었을 때 결과가 다름 (타 트랜잭션의 UPDATE).  
* **Phantom Read:** 없던 데이터가 조회되거나 있던 데이터가 사라짐 (타 트랜잭션의 INSERT/DELETE).  



**3) 실무 적용**  

* 대부분의 상용 RDBMS(Oracle, SQL Server)는 기본적으로 **Read Committed**를 사용하며, MySQL(InnoDB)은 **Repeatable Read**를 기본값으로 사용하되 MVCC(다중 버전 동시성 제어)를 통해 Phantom Read를 방지합니다. 성능과 데이터 무결성 간의 트레이드오프를 고려한 설정이 필요합니다.  

### 2교시형 문제 풀이

## 1. IT 투자성과 평가  

### I. IT 투자성과 평가의 개요  

성공적인 정보화 비즈니스를 위해서는 막대한 자본이 투입되는 IT 투자에 대한 객관적이고 체계적인 성과 측정이 필수적입니다. IT 투자성과 평가는 정보화 사업의 타당성을 검증하고, 시스템 도입 이후의 실질적인 비즈니스 기여도를 정량적/정성적으로 분석하는 활동입니다.  

### II. IT 투자성과 평가의 3대 지표 (가)  

정보시스템의 성과를 다각도로 평가하기 위해 시스템의 내재적 품질, 실제 활용도, 비즈니스 기여도를 아우르는 3가지 지표를 활용합니다.  

| 구분 | 개념 및 목적 | 주요 측정 항목 |
| --- | --- | --- |
| **품질지표 (Quality)** | 구축된 정보시스템 자체의 기술적, 기능적 우수성과 안정성 평가 | 시스템 가동률, 응답 속도, 장애 발생 건수, 데이터 정합성, 보안성 |
| **이용지표 (Usage)** | 시스템이 실제 사용자들에게 얼마나 활발하게 사용되고 있는지 평가 | 일일 로그인 수(DAU), 메뉴별 접속 빈도, 트랜잭션 발생 건수, 사용자 교육 이수율 |
| **효과지표 (Effect)** | 시스템 활용을 통해 얻은 비즈니스적 가치와 재무적/비재무적 성과 평가 | 비용 절감액, 매출 증가율, 업무 처리 시간 단축, 고객 만족도(CSAT), ROI |

### III. 정보화 투자 성과평가 측정 기법 (나)  

IT 투자의 성과를 객관적으로 측정하기 위해 재무적 접근법과 비재무적 접근법을 혼합하여 사용합니다.  

1\. **재무적 측정 기법 (사전/사후 경제성 분석)**  
* **ROI (Return on Investment, 투자수익률):** 총 투자 비용 대비 누적 순수익의 비율을 측정.  
* **NPV (Net Present Value, 순현재가치):** 투자의 결과로 발생하는 현금 유입과 유출을 현재 가치로 할인하여 합산한 금액 ($NPV &gt; 0$ 이면 타당성 있음).  
* **IRR (Internal Rate of Return, 내부수익률):** NPV를 0으로 만드는 할인율.  
* **TCO (Total Cost of Ownership, 총소유비용):** 하드웨어/소프트웨어 도입 비용뿐만 아니라 유지보수, 교육, 운영 비용을 모두 포함한 직접/간접 비용.  


2\. **비재무적/통합적 측정 기법**  
* **IT-BSC (IT Balanced Scorecard):** 조직의 전략과 IT를 연계하여 기업의 성과를 4가지 관점(기업기여도, 사용자 지향, 운영 프로세스, 미래지향)에서 평가하는 프레임워크.  
* **정보화 평가 모형 (Val IT 등):** IT 투자의 비즈니스 가치 창출을 보장하기 위해 포트폴리오 관리, 투자 관리, 가치 실현 관점에서 평가.  



### IV. 결론 및 실무적 제언  

IT 투자는 단순한 시스템 도입을 넘어 비즈니스 혁신의 도구가 되어야 합니다. 따라서 도입 전(사전 타당성), 구축 중(공정/품질 관리), 도입 후(성과 평가)의 전 주기에 걸친 지속적인 라이프사이클 관리가 필요합니다.

## 2. EVM과 Burn Down Chart 및 프로젝트 적용 방안  

### I. 프로젝트 관리 테일러링의 필요성  

최근 비즈니스 환경의 불확실성이 증가하면서 전통적인 폭포수(Waterfall) 기반의 통제 중심 관리와 애자일(Agile) 기반의 유연한 관리를 결합하려는 하이브리드(Hybrid) 접근법이 요구되고 있습니다. 프로젝트 관리자는 각 방법론의 진척 관리 도구인 EVM과 Burn Down Chart의 특성을 이해하고 테일러링해야 합니다.  

### II. EVM과 Burn Down Chart 비교 분석  

| 구분 | EVM (Earned Value Management) | Burn Down Chart |
| --- | --- | --- |
| **기반 방법론** | 전통적 폭포수(Waterfall) 방법론 | 애자일(Agile / Scrum) 방법론 |
| **관리 목적** | 전체 프로젝트의 '비용'과 '일정' 통제 | 스프린트 내 남은 '작업량(시간/포인트)' 시각화 |
| **측정 기준** | 화폐 가치 또는 투입 공수 (Cost) | 스토리 포인트 또는 잔여 시간 (Story Points / Hours) |
| **주요 지표** | PV(계획가치), EV(획득가치), AC(실제원가)<br>

<br>CV, SV, CPI, SPI | 잔여 작업량 트렌드 라인, Ideal Line (이상적 소진 선) |
| **장점** | 프로젝트의 완료 시점 비용/일정 예측 용이 | 팀원 간 직관적인 진척 공유, 즉각적인 이슈 식별 가능 |
| **단점** | 지표 산출을 위한 오버헤드 큼, 잦은 변경에 취약 | 품질이나 실제 비용에 대한 통제력은 다소 부족 |

### III. 프로젝트 적용 방안 (테일러링 전략)  

프로젝트 규모와 요구사항의 불확실성에 따라 **AgileEVM(애자일 기성고 관리)** 형태의 테일러링을 적용할 수 있습니다.  

1\. **포트폴리오 및 프로그램 레벨 (EVM 적용):**  
* 경영진 및 PMO 보고용으로 전체 예산과 마일스톤 관리를 위해 EVM을 사용합니다.  
* 백로그(Backlog) 전체를 기반으로 총 스토리 포인트를 산정하고, 완료된 릴리즈/스프린트 단위로 EV를 환산하여 프로젝트 전체의 CPI(비용성과지수), SPI(일정성과지수)를 산출합니다.  


2\. **실무 실행 레벨 (Burn Down Chart 적용):**  
* 스크럼 팀 내부의 2~4주 단위 스프린트 실행 시에는 Burn Down Chart를 활용합니다.  
* 데일리 스크럼(Daily Scrum)에서 팀원들이 남은 작업을 시각적으로 확인하고, 방해 요소(Impediments)를 즉각 해결하는 데 집중합니다.  



### IV. 결론  

관리 통제(EVM)와 팀의 자율성(Burn Down Chart)을 상호 보완적으로 테일러링하면, 거시적인 예산/일정 준수와 미시적인 요구사항 변화 대응이라는 두 마리 토끼를 모두 잡을 수 있습니다.

## 4. CPU 스케줄링 알고리즘  

### I. CPU 스케줄링의 개요  

다중 프로그래밍 환경에서 CPU의 활용률을 극대화하고, 프로세스의 응답 시간을 최소화하기 위해 Ready Queue에 대기 중인 프로세스 중 어떤 프로세스에 CPU를 할당할지 결정하는 운영체제의 핵심 메커니즘입니다.  

### II. 비선점형(Non-Preemptive) 스케줄링: SJF와 HRN 비교 (가)  

**1. SJF (Shortest Job First)**  

* **개념:** 준비 큐에 있는 프로세스 중 CPU 요구 시간(Burst Time)이 가장 짧은 프로세스에 먼저 CPU를 할당하는 방식입니다.  
* **특징:** 평균 대기 시간을 최소화하는 데 최적화되어 있습니다.  
* **문제점:** 실행 시간이 긴 프로세스는 영원히 CPU를 할당받지 못하는 **기아 현상(Starvation)** 이 발생할 수 있습니다.  

**2. HRN (Highest Response-ratio Next)**  

* **개념:** SJF의 기아 현상을 해결하기 위해 대기 시간을 고려한 **에이징(Aging)** 기법을 적용한 알고리즘입니다.  
* **우선순위 산출 공식:**  

$$Priority = \frac{Waiting\&#95;Time + Service\&#95;Time}{Service\&#95;Time}$$  


* **특징:** 대기 시간이 길어질수록 우선순위가 높아지므로, 긴 작업도 결국에는 CPU를 할당받아 기아 현상을 방지합니다.  

### III. 선점형(Preemptive) 스케줄링: MLQ와 MLFQ 비교 (나)  

**1. MLQ (Multi Level Queue, 다단계 큐)**  

* **개념:** 프로세스의 중요도(예: 시스템, 대화형, 배치)에 따라 우선순위가 다른 여러 개의 고정된 큐를 구성하고, 각 큐마다 다른 스케줄링 알고리즘(RR, FCFS 등)을 적용합니다.  
* **특징:** 프로세스는 한 큐에 영구적으로 할당되며, 큐들 간의 이동이 불가합니다.  
* **단점:** 하위 우선순위 큐에 있는 프로세스는 기아 현상을 겪을 수 있으며, 유연성이 부족합니다.  

**2. MLFQ (Multi Level Feedback Queue, 다단계 피드백 큐)**  

* **개념:** MLQ의 단점을 보완하여, 프로세스의 실행 패턴에 따라 프로세스가 **큐 사이를 동적으로 이동(Feedback)** 할 수 있도록 설계된 알고리즘입니다.  
* **특징:**  
* CPU 버스트가 긴 프로세스는 점차 하위 큐로 강등되어 Time Quantum이 커집니다.  
* 하위 큐에서 너무 오래 대기한 프로세스는 상위 큐로 승격(Aging)시켜 기아 현상을 방지합니다.  
* 현대 범용 운영체제에서 가장 많이 사용되는 방식입니다.  



### IV. 결론  

시스템의 목적(처리량 극대화, 응답시간 최소화 등)에 따라 적절한 스케줄링 방식이 요구되며, 최신 OS는 복잡한 워크로드를 효율적으로 처리하기 위해 MLFQ와 같은 동적이고 적응적인 방식을 채택하고 있습니다.

## 5. 소프트웨어 3R과 리팩토링 (유지보수 관점)  

### I. 노후화된 정보시스템 유지보수를 위한 SW 공학적 접근  

시스템 도입 후 운영 단계가 길어짐에 따라 'Lehman의 소프트웨어 변화 원리'처럼 시스템은 지속적으로 복잡해지고 유지보수 비용(기술 부채)이 급증합니다. 이를 해결하기 위해 소스코드의 구조를 개선하는 리팩토링과, 시스템 자산을 재활용하고 진화시키는 3R 기법이 필수적입니다.  

### II. 리팩토링(Refactoring)의 개념 및 활용  

* **개념:** 겉으로 보이는 소프트웨어의 동작(기능)은 변경하지 않으면서, 내부 구조를 개선하여 코드의 가독성을 높이고 유지보수성을 향상시키는 기법입니다.  
* **유지보수 관점의 활용:** 중복 코드 제거(Extract Method), 거대 클래스 분리, 의미 없는 변수명 변경 등을 통해 기술 부채(Technical Debt)를 청산하고 향후 확장을 용이하게 합니다.  

### III. 소프트웨어 3R (Reverse-Engineering, Re-Engineering, Re-use)  

운영 중인 시스템의 수명을 연장하고 가치를 극대화하기 위한 세 가지 기법입니다.  

| 구분 | 개념 (정의) | 운영 및 유지보수 관점의 기대효과 및 특징 |
| --- | --- | --- |
| **역공학 (Reverse-Engineering)** | 완성된 소스코드나 실행 파일(목적 코드)을 분석하여, 설계 문서, 아키텍처, 요구사항 등의 **상위 레벨 추상화 정보를 추출**하는 과정 | - 문서가 유실된 레거시 시스템의 구조 파악<br>

<br>- 코드 내 숨겨진 비즈니스 룰 및 로직 추출 |
| **재공학 (Re-Engineering)** | 기존 시스템의 기능은 유지하면서, 새로운 요구사항이나 최신 기술 환경에 맞게 **소프트웨어의 전체 구조를 뜯어고치고 재구축**하는 과정 (역공학 + 리팩토링 + 순공학) | - 낡은 언어/플랫폼(예: COBOL → Java) 마이그레이션<br>

<br>- 시스템 수명 연장 및 유지보수성 획기적 향상 |
| **재사용 (Re-use)** | 이미 개발되어 검증된 소프트웨어 자산(함수, 컴포넌트, 아키텍처, 프레임워크)을 **새로운 시스템 개발이나 유지보수 작업에 다시 사용**하는 과정 | - 코드 재작성 방지로 결함률 감소<br>

<br>- 유지보수 개발 기간 단축 및 품질 향상 |

### IV. 결론  

유지보수 단계에서 발생하는 코드 스파게티화와 문서 불일치 문제는 3R과 리팩토링을 통해 극복할 수 있습니다. 특히 MSA(Microservices Architecture)로의 전환 등 레거시 모더나이제이션(Modernization) 과정에서 이 기법들은 핵심적인 역할을 수행합니다.

## 6. ISMP (Information System Master Plan)  

### I. ISMP의 개요 (가)  

* **정의:** 특정 소프트웨어 개발 사업에 대한 상세 분석과 요건 정의를 수행하여, 시스템 구축을 위한 **명확한 제안요청서(RFP)를 도출**하기 위한 정보시스템 종합 구축 계획입니다.  
* **목적:** 발주자(요구사항 불명확)와 수주자(범위 크리프 현상) 간의 분쟁을 예방하고, 소프트웨어 사업의 예산 산정 타당성을 확보하며, 상세 요구사항 기반의 정확한 발주 체계를 확립하기 위함입니다.  

### II. ISMP 수행방법론 체계와 절차 (나)  

ISMP는 정보화 전략부터 구체적인 실행 계획까지 5단계 절차로 수행됩니다.  

1\. **프로젝트 착수 및 참여자 결정:** 프로젝트 범위 정의, TFT 구성 및 이해관계자 인터뷰 계획 수립.  
2\. **정보시스템 방향성 수립:** 비즈니스/IT 환경 분석(As-Is), 선진사례 벤치마킹을 통한 시사점 도출 및 시스템 구축 방향성(To-Be) 설정.  
3\. **업무 및 정보기술 요건 분석:** 현행 업무 프로세스 분석, 신규 요구사항 수집 및 상세 요건(기능/비기능/데이터) 정의.  
4\. **정보시스템 구조 및 요건 정의:** To-Be 애플리케이션, 데이터, 기술, 보안 아키텍처 초안 설계 및 상세 요구사항 명세서(RTM) 작성.  
5\. **정보시스템 구축사업 타당성 분석 및 실행계획 수립:** 소요 예산 산정(기능점수 기반), ROI/비용편익 분석, 사업 추진 로드맵 수립 및 **제안요청서(RFP) 작성**.  

### III. ISP, EA/ITA, ISMP 비교 (다)  

| 구분 | ISP (Information Strategy Planning) | EA / ITA (Enterprise Architecture) | ISMP (Information System Master Plan) |
| --- | --- | --- | --- |
| **목적** | 경영전략과 연계된 전사적 정보화 전략 및 중장기 로드맵 수립 | 조직의 비즈니스와 IT 인프라를 체계적으로 분류한 청사진/프레임워크 구축 | **특정 사업/시스템 구축**을 위한 상세 요건 정의 및 발주(RFP) 준비 |
| **수준/범위** | 전사적 (Enterprise), 전략적 (Strategic) | 전사적 (Enterprise), 거시적/구조적 | **특정 시스템 (System)**, 전술적 (Tactical) |
| **핵심 산출물** | 중장기 정보화 과제 및 마스터플랜 | 아키텍처 매트릭스, 참조 모델 | 상세 요구사항 명세서, 예산 내역, **RFP** |

### IV. 결론  

성공적인 공공/민간 IT 프로젝트를 위해서는 선(先) ISP를 통한 방향성 수립 후, 구체적인 사업 착수 전에 ISMP를 수행하여 요구사항을 명확히 하는 2단계 접근이 프로젝트 실패 리스크를 줄이는 핵심 방안입니다.

## 7. 소프트웨어 안전성 분석 개념 및 분석 기법  

### I. 소프트웨어 중심 사회에서의 SW 안전성 중요성  

자동차, 항공, 철도, 의료기기 등 사이버-물리 시스템(CPS)에서 소프트웨어의 비중이 커지면서, SW 결함이 인간의 생명, 재산, 환경에 치명적인 사고로 직결될 수 있습니다. 이에 따라 소프트웨어 안전성 확보가 시스템 설계의 최우선 과제로 대두되었습니다.  

### II. 소프트웨어 안전성(Safety) 개념  

* **개념:** 소프트웨어가 시스템 내에서 실행될 때, 수용 불가능한 위험(Unacceptable Risk)이나 재난(사상자 발생, 심각한 기기 파손 등)을 초래하지 않도록 보장하는 속성입니다.  
* **핵심 원리:**  
* **Fail-Safe:** 고장 발생 시 시스템을 안전한 상태로 유도.  
* **Fault Tolerance (결함 허용):** 일부 결함이 발생해도 시스템이 본연의 기능을 지속하도록 다중화(Redundancy) 적용.  



### III. 소프트웨어 안전성 분석 기법  

잠재적인 위험(Hazard)을 식별하고, 원인과 결과를 분석하는 방법론입니다.  

| 분석 기법 | 접근 방식 | 세부 설명 및 특징 |
| --- | --- | --- |
| **FMEA**<br>

<br>(Failure Mode and Effects Analysis) | Bottom-Up<br>

<br>(상향식) | 부품이나 모듈 레벨의 개별 결함(Failure Mode)이 전체 시스템(Effects)에 미치는 영향을 분석. 치명도(Criticality)를 평가하는 FMECA로 확장 가능. |
| **FTA**<br>

<br>(Fault Tree Analysis) | Top-Down<br>

<br>(하향식) | 시스템 레벨의 치명적인 사고(Top Event)를 먼저 정의하고, 연역적 논리 게이트(AND/OR)를 사용하여 원인이 되는 하위 결함 트리를 도출하는 기법. |
| **HAZOP**<br>

<br>(Hazard and Operability Study) | Guideword<br>

<br>(가이드워드 기반) | 설계 의도에서 벗어난 상태를 식별하기 위해 'No, More, Less, As well as' 등의 가이드워드를 사용하여 프로세스의 위험 요소를 브레인스토밍하는 기법. |
| **STPA**<br>

<br>(Systems-Theoretic Process Analysis) | System Control<br>

<br>(제어 구조 기반) | 시스템을 복잡한 제어 루프(Control Loop)로 모델링하고, 컴포넌트 간의 잘못된 상호작용이나 통제 결함으로 인해 발생하는 위험을 분석하는 최신 기법 (자율주행 등에 적합). |

### IV. 결론  

안전 최우선(Safety-Critical) 시스템의 무결성을 확보하기 위해서는 ISO 26262(자동차), DO-178C(항공) 등 산업별 국제 표준을 준수하고, 요구사항 단계부터 STPA와 FTA를 결합한 철저한 안전성 분석과 내재화가 수행되어야 합니다.

## 8. 빅데이터 플랫폼을 위한 LOD (Linked Open Data) 기술 요소  

### I. 개방형 데이터 생태계 확장을 위한 LOD 개요  

빅데이터 환경에서 공공 및 민간의 데이터 가치를 극대화하기 위해서는 기계가 이해하고 데이터를 넘나들며 연결할 수 있는 웹 기반의 지식 네트워크가 필요합니다. LOD(Linked Open Data)는 팀 버너스리가 제안한 시맨틱 웹(Semantic Web)의 Linked Data 원칙과 누구나 사용할 수 있는 Open Data의 개념이 결합된 개방형 데이터 발행 패러다임입니다.  

### II. LOD의 4대 원칙 (팀 버너스리)  

1\. **URI(Uniform Resource Identifier) 사용:** 사물이나 개념에 대해 고유한 식별자인 URI를 부여합니다.  
2\. **HTTP URI 사용:** 사람들이 해당 식별자를 통해 웹에서 데이터를 조회(Dereferencing)할 수 있게 합니다.  
3\. **표준 기술 활용 (RDF, SPARQL):** 조회 시 기계가 읽을 수 있는 표준 형식(RDF)으로 유용한 정보를 제공합니다.  
4\. **링크 제공:** 더 많은 정보를 탐색할 수 있도록 다른 데이터의 URI와 연결(Link)합니다.  

### III. LOD 플랫폼 구성을 위한 핵심 기술요소  

| 계층 | 기술 요소 | 기능 및 설명 |
| --- | --- | --- |
| **식별 체계** | **URI / IRI** | 인터넷 상의 논리적/물리적 자원을 고유하게 식별하는 명명 규약 (IRI는 다국어 지원). |
| **데이터 표현** | **RDF (Resource Description Framework)** | 웹 상의 자원과 그들 간의 관계를 **주어(Subject), 동사(Predicate), 목적어(Object)의 트리플(Triple)** 형태로 정의하는 데이터 모델. |
| **어휘 및 구조** | **Ontology / OWL** | 자원 간의 계층 구조, 속성, 관계 등을 명시적으로 정의하여 기계가 데이터를 추론할 수 있게 하는 어휘(Vocabulary) 정의 언어. |
| **질의 및 검색** | **SPARQL** | RDF 형식으로 저장된 트리플 데이터를 질의하고 검색, 조작하기 위한 시맨틱 웹 표준 쿼리 언어 (SQL과 유사한 문법). |
| **접근 및 연계** | **LOD Cloud / End-point** | 외부 시스템이 SPARQL 질의를 던져 데이터를 가져갈 수 있도록 제공하는 인터페이스(SPARQL Endpoint) 및 연결망. |

### IV. 결론  

빅데이터 분석 플랫폼에서 LOD 기술을 적용하면 사일로(Silo)화된 데이터 섬을 연결하여 거대한 지식 그래프(Knowledge Graph)를 구축할 수 있습니다. 이를 통해 인공지능의 추론 능력을 강화하고 데이터 기반의 새로운 융합 비즈니스 생태계를 창출할 수 있습니다.
