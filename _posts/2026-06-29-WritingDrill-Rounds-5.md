### [감리] 운영감리 (OP)

**[정의]**

- 개발완료후 인도된 정보시스템에 대해 유지보수를 제외한 운영기획 및 관리, 모니터링, 테스트, 사용자 지원을 포함한 정보시스템의 정상적 운영에 필요한 제반활동

'- 전자정부법 시행령 제71조 기준  (시스템의 정상적 운영과 서비스 제공의 적정성에 초점)

**[핵심/키워드]**

* 개발SW : 릴리즈 및 배포관리, 테스트지원, 장애관리
* 인프라 : SLA관리, 서비스보고, 연속성 및 가용성관리, 용량관리, 정보보안관리, 공급자관리, 운영상태관리

### [감리] 유지보수감리(MA)

**[정의]**

- 정보시스템 개발, 구축 완료후 기능변경, 추가, 보완, 폐기, 사용방법의 개선, 문서보완 등의 정보시스템 개선에 필요한 제반 활동

- 전자정부법 시행령 제71조 기준 (시스템 기능 개선과 문제 해결, 지속적인 유지관리의 적정성에 중점)

**[핵심/키워드]**

* 개발SW : 응용서비스 모니터링, 장애처리, 사용자지원, 성능관리, 정기/비정기점검, 유지보수계획, 요구사항관리, 유지보수이행
* 상용SW : 유지보수계획, 표준및절차, 업그레이드 및 패치, 이전 및 재설치, 일상지원, 운영자/사용자교육, 보안조치
* 인프라 : OS/HW업그레이드, 긴급/장애처리, 부품지원, 운영상태관리, 이전 및 재설치

### [감리] 상주감리

**[정의]**

정보시스템 구축 사업 현장에 상주하며, 사업의 위험도나 난이도가 높다고 판단될 때 수행되는 감리 활동

**[핵심/키워드]**

* PMO -  프로젝트 수행을 지원·통제하는 내부 관리 조직
* 상주감리 -  외부 독립적 입장에서 품질과 절차의 준수 여부를 상시 점검하는 역할

### [감리] 생성형 인공지능(AI) 정보시스템의 데이터 품질 감리 점검 방안

**[정의]**

인공지능 학습용 데이터 품질관리 가이드라인에서 제시한 데이터 품질관리 활동에 대해서, 구축 절차와 절차 별로 생성되는 산출물, 성과의 3가지 감리 관점 및 기준을 제시하는 방법론

**[핵심/키워드]**

* 절차 : 구축계획수립, 데이터 획득/수집, 정제, 가공, 학습 
(품질관리계획서, 가공데이터&품질관리보고서, 데이터 정제도구, 가공데이터, AI모델 설명서)

### [감리] 3단계 감리

**[정의]**

**[핵심/키워드]**

* 요구정의 단계 감리 :  요구사항정의서가 계약서의 과업 내용을 적정하게 반영
* 설계 단계 감리 :  설계 결과물이 요구사항정의서 및 관련 기준에 부합하는지 검토하고 확인
* 종료 단계 감리 : 시스템 개발 완료 후 전반적인 품질, 안전성, 적합성을 최종 점검

### [소공] 리팩토링(Refactoring)

**[정의]**

- SW 외부기능 변경없이 내부 로직만을 수정하여 유지보수 및 개발 생산성 향상 기법

**[핵심/키워드]**

[ 결합도 측면 ]
- Move Method/Field: 타 클래스와 결합도가 높은 메소드/속성 이전 
- Extract Class: 여러 기능이 결합된 클래스는 개별 클래스로 분리
[ 응집도 측면 ] 
- Pull Up Method: 서브클래스에 같은 결과를 반환하면 슈퍼 클래스로 이전
- Pull Up Field: 서브클래스와 같은 속성을 사용하면 슈퍼 클래스로 이전 
- Push Down Method: 슈퍼클래스의 메소드가 서브 클래스만 사용한다면 서브클래스로 이동 
- Push Down Field: 슈퍼클래스의 속성이 서브 클래스만 사용한다면 서브클래스로 이동 
- Inline Class: 클래스가 하는 일이 많지 않은경우 다른 클래스로 옮기고 제거
[ 가독성 측면 ]
- Rename: 클래스나 메소드의 동작을 알수있도록 이름 변경

*객체의 동일 인터페이스로 다양한 행위를 구현하는 핵심 개념
1) 오버로딩 (Overloading) 
- 동일 클래스, 같은 이름의 오퍼레이션, 이름은 같고 파라미터의 개수와 형식은 다름

2) 오버라이딩 (Overriding) 
- 부모의 상속받은 오퍼레이션 자식 클래스가 같은 이름으로 재정의, 파라미터의 개수와 타입 동일

### [UML] 정보은닉 (Information Hiding)

**[정의]**

메시지만으로 객체와 상호작용하도록 정보를 숨겨서 접근을 제한하는 특징

* 정보 보안성

**[핵심/키워드]**

[ 함수 ]
- Setter : 정보 세팅 함수
- Getter : 정보 조회 함수

### [UML] 전략 패턴 (Strategy Pattern)

**[정의]**

동일한 기능을 수행하는 다양한 알고리즘을 인터페이스로 캡슐화해 실행 시점에 동적으로 교체 가능한 설계 패턴

**[핵심/키워드]**

*알고리즘을 캡슐화해 유연하게 교체 가능한 행동 패턴
* 특징 : 다형성
* 구성 : strategy, concretestrategy, context
* 사례 : threadpool, compare()
* 패턴 : template, flyweight에서 사용됨

### [UML] 옵저버 패턴 (Observer Pattern)

**[정의]**

하나의 객체 상태 변경을 의존 객체들에게 자동 통보하여 일관된 동작을 유지하도록 하는 행위적 설계 패턴

**[핵심/키워드]**

*이벤트 기반 아키텍처·분산 시스템·IoT 등에서 핵심 구조로 활용
* 특징 : 상태 전달 및 갱신
* 구성 : subject, observer, concreteobserver
* 사례 : IObserver&lt;T&gt;, delegate, swing
* 패턴 : mediator를 이용 가능

### [UML] 프록시 패턴 (Proxy Pattern)

**[정의]**

접근 대상 객체와 동일한 인터페이스를 제공하는 대리인 객체를 이용해 타겟 객체 접근 전에 추가적인 작업의 기회를 제공하는 패턴

**[핵심/키워드]**

* 객체 접근을 간접화하여 제어·보안·성능을 향상시키는 구조적 설계 패턴
* 특징 : 대리자, 연결
* 구성 : 서브젝트, 리드서브젝트, 프록시
* 종류 : normal, virtual, protect, logging, remote, caching
* 사례 : invoke()

### [UML] 4+1 Architectural View Model

**[정의]**

고객 요구사항을 중심으로 설계자, 시스템통합자, 개발자, 시스템엔지니어의 관점으로 SW Architecture를 구축하는 설계방법

**[핵심/키워드]**

* 특징 : 이해관계자, 4가지 관점
* 구성 : usecase, logical, process, development, physical

### [UML] 클래스 다이어그램 (Class diagram)

**[정의]**

클래스간의 속성, 관계, 메서드를 정의하여, 객체지향의 기본 구조를 표현하는 다이어그램

**[핵심/키워드]**

* 특징 : 클래스, 정적, 구조적
* 요소 : 클래스, 속성, 메서드, 접근성, 관계

### [UML] 활동 다이어그램 (Activity Diagram)

**[정의]**

시스템 혹은 비즈니스 프로세스나 워크플로우의 흐름을 순차적으로 표현하는 다이어그램

[ 관점 ]
행동 (Behavioral)
동적 (Dynamic)

**[핵심/키워드]**

[ 구성요소 ]
시작상태 (Initial State), 종료상태 (Final State), 액션상태 (Action State), 전이 (Transition), 분기 (Decision), 구획면 (Swin lane), 시그널 (Signal), 동기바 (Synchronization)

[ 주요 흐름 ]
제어 흐름 (Control Flow), 객체/데이터 흐름 (Object Flow)

### [UML] 미디에이터 패턴 (Mediator Pattern)

**[정의]**

각 객체가 관련성을 갖는 다른 객체에 대한 참조 관계를 독립된 다른 객체 한 곳(Mediator)으로 몰아서 관리를 용이하게 해주는 패턴

**[핵심/키워드]**

* 특징 : 통합관리
* 구성 : mediator, concretemediator, colleague
* 사례 : mediateR
* 패턴 : observer 패턴에서 사용됨

### [UML] Factory Method Pattern

**[정의]**

객체의 생성 과정을 서브 클래스에서 정의하도록 위임하는 생성패턴

* 상위 클래스 : Factory Method, 하위 클래스 : 구체적 객체 생성

**[핵심/키워드]**

[ 구성요소 ]
Product : 생성 객체 인터페이스 정의 
ConcreteProduct : 인터페이스 구현
Creator : Product 타입의 객체 반환 팩토리 메소드 선언(부모)
ConcreteCreator : 인스턴스 반환 (자식)
Factory Method : 객체 생성 위임 메서드
[ 연관 패턴 ]
Template : 생성 시 Factory 이용
Abstract Factory : 실제 구현 시 Factory 이용

### [UML] 서킷 브레이커(Circuit Breaker) 패턴

**[정의]**

- 장애를 감지하고 예상치 못한 시스템 장애가 지속적으로 반복되는 것을 방지하는 로직을 캡슐화한 디자인 패턴

**[핵심/키워드]**

* 회로상태
Open(오류상태)
Close(정상상태)
Half-open(지속시간 이후)

### [UML] 타이밍 다이어그램 (Timing Diagram)

**[정의]**

시간의 흐름에 따라 객체의 상태 변화와 이벤트 발생을 시각적으로 표현한 다이어그램

**[핵심/키워드]**

* 특징 : 시퀀스 + 상태, 동적, 상호작용

### [UML] 객체 다이어그램 (Object Diagram)

**[정의]**

특정 시점에서 객체의 상태 및 관계를 나타내어 클래스 다이어그램의 인스턴스를 표현하는 다이어그램

**[핵심/키워드]**

* 특징 : 특정시점, 구조적, 정적
* 구성 : 객체(object), 연결(link)

### [UML] 퍼사드 패턴 (Facade Pattern)

**[정의]**

클래스 라이브러리 같은 어떤 소프트웨어의 다른 커다란 코드 부분에 대한 간략화 된 인터페이스를 제공하는 디자인 패턴

**[핵심/키워드]**

* 특징 : 라이브러리, API집합
* 구성 : 퍼사드, 클라이언트, 패키지
* 사례 : system.data.sqlclient
* 패턴 : 싱글톤 패턴 이용 가능

### [UML] 디자인 패턴 (Design Pattern)

**[정의]**

반복적으로 나타나는 문제를 해결해온 전문가들의 경험을 모아서 정리한 모음집

**[핵심/키워드]**

* 특징 : 문제해결 BP
* 구성 : 이름, 문제, 해법, 결과
* 원칙 : 캡슐화, 위임, 인터페이스, 느슨한결합, 개방 및 폐쇄, 의존관계
* 유형 : 생성, 구조, 행위

### [UML] 유즈케이스 다이어그램 (Usecase diagram)

**[정의]**

사용자(액터)와 시스템간의 주요 기능과 상호작용으로 요구사항을 표현하는 다이어그램

**[핵심/키워드]**

* 특징 : 사용자 요구사항, 행동적, 동적
* 구성 : 액터, 유즈케이스, 패키지, 스테레오타입, 관계

### [UML] 프로토타입 패턴 (Prototype Pattern)

**[정의]**

생성할 객체의 종류를 명시하고, 이렇게 만든 견본을 복사해서 새로운 객체를 생성하는  패턴

**[핵심/키워드]**

* 특징 : 원본 복사
* 구성 : 프로토타입, 콘크리트프로토타입, 클라이언트
* 사례 : java.lang.cloneable
* 패턴 : 추상팩토리 패턴 이용

### [UML] 싱글톤 패턴 (Singleton Pattern)

**[정의]**

오직 단 하나의 인스턴스만을 생성하고 전역변수 사용과 같이 어디서나 인스턴스에 접근 가능한 디자인 패턴

**[핵심/키워드]**

* 특징 : 오직 하나 인스턴스
* 사례 : getinstance()
* 구현 : eager, lazy, double checked
* 패턴 : 추상팩토리, 퍼사드 패턴 이용 가능
