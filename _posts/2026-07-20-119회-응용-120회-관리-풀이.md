### 1교시형 문제 풀이

## 1. SVM (Support Vector Machine)

### I. SVM의 개요

기계학습 분야의 패턴 인식 및 자료 분석을 위한 지도학습 모델로, 데이터 집합을 분류하는 여러 선형 평면 중에서 여백(Margin)을 최대화하는 최적의 초평면(Hyperplane)을 찾아 미지의 데이터를 분류하는 알고리즘입니다.  

### II. SVM의 핵심 구성요소

* **초평면 (Hyperplane):** 서로 다른 클래스의 데이터를 분할하는 결정 경계(Decision Boundary)  
* **서포트 벡터 (Support Vector):** 초평면에 가장 가까이 위치한 각 클래스의 데이터 포인트  
* **여백 (Margin):** 초평면과 서포트 벡터 간의 수직 거리 (Margin이 클수록 일반화 성능 향상)  
* **커널 트릭 (Kernel Trick):** 선형으로 분리할 수 없는 데이터를 고차원 공간으로 매핑하여 비선형 분류를 가능하게 하는 기법 (다항식 커널, RBF 커널 등)  

### III. SVM의 특징

* **장점:** 고차원 데이터 및 비선형 데이터 처리에 강함, 과적합(Overfitting) 위험이 낮음  
* **단점:** 데이터 세트가 클 경우 학습 속도가 느려짐, 최적의 커널 함수 및 파라미터 선택이 어려움

## 2. 인공신경망의 오류 역전파(Backpropagation) 알고리즘

### I. 오류 역전파의 개요

인공신경망 학습 시, 출력층에서 발생한 오차(Loss)를 은닉층을 거쳐 입력층 방향으로 역전파하면서 각 노드의 가중치(Weight)를 업데이트하는 지도학습 알고리즘입니다.  

### II. 오류 역전파의 학습 메커니즘

1\. **순전파 (Forward Propagation):** 입력 데이터를 통해 각 층의 활성화 함수를 거쳐 최종 출력값(예측값) 계산  
2\. **오차 계산 (Loss Calculation):** 실제 정답값과 예측값의 차이인 손실 함수(Loss Function) 계산  
3\. **역전파 (Backward Propagation):** 연쇄 법칙(Chain Rule, $\frac{\partial L}{\partial w}$)을 활용하여 출력층에서 입력층으로 각 가중치의 오차 기울기(Gradient) 계산  
4\. **가중치 갱신 (Weight Update):** 경사하강법(Gradient Descent)을 사용하여 오차를 최소화하는 방향으로 가중치 업데이트  

### III. 오류 역전파의 한계점과 극복 방안

* **기울기 소실 (Vanishing Gradient):** 신경망이 깊어질수록 역방향 전파 시 기울기가 0으로 수렴하여 가중치가 업데이트되지 않는 현상  
* **해결책:** ReLU(Rectified Linear Unit) 활성화 함수 사용, 배치 정규화(Batch Normalization), ResNet(잔차 연결) 활용

## 3. 오버라이딩(Overriding)으로 함수를 재정의 하는 이유

### I. 오버라이딩의 개요

객체지향 프로그래밍(OOP)에서 상위 클래스(부모)가 정의한 메서드를 하위 클래스(자식)가 상속받아 자신의 목적에 맞게 동일한 시그니처(이름, 매개변수)로 재정의하는 기법입니다.  

### II. 함수를 재정의(Overriding)하는 이유 (핵심 목적)

* **다형성(Polymorphism) 구현:** 동일한 메시지(메서드 호출)에 대해 객체의 타입에 따라 서로 다른 고유한 동작을 수행하도록 보장합니다.  
* **코드 재사용성 및 확장성 향상:** 상위 클래스의 기본 기능을 유지하면서, 하위 클래스에서 필요한 기능만 수정하여 개방-폐쇄 원칙(OCP)을 준수할 수 있습니다.  
* **동적 바인딩(Dynamic Binding) 지원:** 컴파일 타임이 아닌 런타임에 참조하는 실제 객체의 타입에 맞는 메서드가 호출되어 유연한 객체 관리를 가능하게 합니다.  

### III. 오버라이딩과 오버로딩 비교

| 구분 | 오버라이딩 (Overriding) | 오버로딩 (Overloading) |
| --- | --- | --- |
| **적용 범위** | 상속 관계 (부모-자식 간) | 동일 클래스 내부 |
| **메서드 서명** | 이름, 매개변수 모두 동일 | 이름 동일, 매개변수 다름 |
| **목적** | 다형성 확보, 기능 재정의 | 가독성 증가, 편의성 제공 |

## 4. HTTP 3.0

### I. HTTP 3.0의 개요

기존 TCP 기반의 HTTP/2의 한계(HOL Blocking)를 극복하기 위해, 구글이 개발한 UDP 기반의 QUIC 프로토콜을 전송 계층으로 채택하여 지연 시간을 최소화하고 보안을 내재화한 차세대 웹 통신 프로토콜입니다.  

### II. HTTP 3.0의 주요 기술 및 특징

* **QUIC 프로토콜 채택:** 신뢰성 있는 연결을 위해 TCP 대신 UDP 위에서 재전송, 혼잡 제어를 소프트웨어적으로 구현  
* **0-RTT 연결 설정:** 이전에 통신한 세션 정보를 캐싱하여 핸드쉐이크 과정 없이 즉시 데이터 전송 (Zero Round Trip Time)  
* **독립적 스트림 처리:** 패킷 하나가 손실되어도 다른 스트림에 영향을 주지 않아 TCP의 HOL(Head of Line) Blocking 문제 원천 해결  
* **보안 내재화:** TLS 1.3 계층이 QUIC 내부에 통합되어 암호화가 강제됨  

### III. HTTP 발전 과정 요약

* **HTTP 1.1:** 텍스트 기반, Keep-Alive, Pipelining (HOL 발생)  
* **HTTP 2.0:** TCP 기반, 멀티플렉싱, 서버 푸시, 헤더 압축  
* **HTTP 3.0:** UDP(QUIC) 기반, 0-RTT, 향상된 혼잡 제어 및 보안

## 5. 모니터(Monitor) 프로세스 동기화 기법

### I. 모니터의 개요

다중 프로세스(스레드) 환경에서 공유 자원에 대한 상호배제(Mutual Exclusion)를 프로그래밍 언어 수준에서 안전하게 캡슐화하여 제공하는 고수준 동기화 구조체입니다. (예: Java의 `synchronized` 블록)  

### II. 모니터의 주요 구성요소

* **공유 데이터 (Shared Data):** 여러 스레드가 접근하려는 자원 (모니터 내부의 프로시저를 통해서만 접근 가능)  
* **상호배제 큐 (Entry Queue):** 진입 구역으로, 한 번에 오직 하나의 스레드만 모니터 내부에 진입할 수 있도록 대기하는 큐  
* **조건 변수 (Condition Variable):** 자원이 부족할 때 대기 상태를 관리하기 위한 변수 (`wait()`, `signal()`, `notify()` 연산 제공)  

### III. 모니터와 세마포어(Semaphore) 비교

| 구분 | 모니터 (Monitor) | 세마포어 (Semaphore) |
| --- | --- | --- |
| **추상화 수준** | 고수준 (언어에서 제공) | 저수준 (OS에서 제공) |
| **사용 편의성** | 쉽고 안전함 (자동 락 해제) | 어려움 (개발자 실수로 데드락 발생 가능) |
| **동작 방식** | 캡슐화 및 상호배제 강제 | 변수값(P, V 연산) 기반의 권한 관리 |

## 6. Ad-hoc Network Routing Protocol

### I. Ad-hoc Network Routing Protocol의 개요

기지국(AP) 등 고정된 인프라 없이 모바일 노드(단말기)들만으로 자율적으로 구성된 무선 네트워크 환경에서, 노드 이동으로 인한 동적 토폴로지 변화에 적응하여 최적의 통신 경로를 설정하고 유지하는 규약입니다.  

### II. 라우팅 프로토콜의 분류

* **Table-Driven (Proactive) 프로토콜:** 토폴로지 변화에 상관없이 주기적으로 라우팅 정보를 교환하여 항상 최신 경로표를 유지 (예: DSDV, OLSR)  
* *특징:* 지연 시간이 짧으나 제어 오버헤드가 크고 배터리 소모가 심함  


* **On-Demand (Reactive) 프로토콜:** 데이터 전송 요구가 있을 때만 목적지까지의 경로를 탐색하여 설정 (예: AODV, DSR)  
* *특징:* 대역폭 낭비가 적으나 초기 경로 설정 지연(Latency)이 발생  


* **Hybrid 프로토콜:** 네트워크를 구역(Zone)으로 나누어 내부 구역은 Table-driven, 외부 구역은 On-demand 방식을 혼용 (예: ZRP)  

### III. 활용 분야

군사 전술망, 재난 구조망, 차량 간 통신 네트워크(VANET), IoT 센서 네트워크

## 7. OAuth 2.0

### I. OAuth 2.0의 개요

사용자의 아이디나 비밀번호를 노출하지 않고, 신뢰할 수 있는 서드파티 애플리케이션(Client)이 사용자를 대신하여 특정 자원(API)에 접근할 수 있도록 접근 권한을 위임하는 개방형 인가(Authorization) 표준 프로토콜입니다.  

### II. OAuth 2.0 주요 구성요소 및 토큰

* **Resource Owner:** 보호된 자원의 소유자 (일반 사용자)  
* **Client:** 사용자를 대신해 보호된 자원에 접근하려는 애플리케이션  
* **Authorization Server:** 사용자를 인증하고 Client에게 토큰을 발급하는 서버 (예: 구글/카카오 로그인 서버)  
* **Resource Server:** 사용자의 자원을 호스팅하는 API 서버  
* **Access Token:** 자원 접근을 위한 일회성/단기 권한 증명서 (탈취 방지를 위해 수명이 짧음)  
* **Refresh Token:** Access Token 만료 시 재발급받기 위한 긴 수명의 토큰  

### III. 주요 인가 방식 (Grant Types)

* **Authorization Code Grant:** 인가 코드(Code)를 먼저 받고 이를 통해 토큰을 교환하는 방식 (가장 안전하며 널리 쓰임)  
* **Implicit Grant:** 코드 교환 없이 바로 토큰을 발급 (웹 브라우저 기반 SPA에 주로 사용되나 보안상 점차 지양됨)  
* **Client Credentials Grant:** 클라이언트 자체가 권한을 가질 때 사용 (B2B API 연동)

## 8. 앰비언트 커머스 (Ambient Commerce)

### I. 앰비언트 커머스의 개요

IoT 센서, 인공지능, 컴퓨터 비전 기술을 활용하여 일상 공간(Ambient) 속에서 소비자가 별도의 결제 행위나 모바일 앱 조작 없이 자연스럽고 매끄럽게 쇼핑과 결제를 경험할 수 있도록 하는 차세대 무각성 상거래 형태입니다.  

### II. 핵심 기술 및 적용 사례

* **핵심 기술:**  
* 컴퓨터 비전(객체 추적 및 상품 인식)  
* 센서 퓨전 (선반의 무게 센서, 라이다 등 데이터 융합)  
* 생체 인식 및 위치 추적 기술  
* 엣지 컴퓨팅 (지연 없는 실시간 영상 데이터 처리)  


* **대표 사례:** 아마존 고(Amazon Go)의 **Just Walk Out** 기술 (상품을 들고 매장을 나가면 자동으로 계좌 청구)  

### III. 상거래의 발전 흐름과 시사점

* **발전 방향:** E-Commerce(PC) → M-Commerce(모바일) → V-Commerce(음성/AI 스피커) → **Ambient Commerce(무각성/센서)**  
* **시사점:** 마찰 없는 쇼핑(Frictionless Shopping)으로 극강의 소비자 편의를 제공하지만, 매장 내 다수의 카메라 및 센서로 인한 개인정보 침해 및 감시 우려(프라이버시 이슈)에 대한 법적/기술적 가이드라인 마련이 필수적입니다.  

### 2교시형 문제 풀이

## 1. VPN 구현 방식 및 서비스 형태 비교와 SSL VPN

### I. VPN(Virtual Private Network)의 개요

공중망(Public Network)을 전용선(Private Network)처럼 사용할 수 있게 하여, 원격지 사용자나 지사 간의 안전한 데이터 통신을 경제적으로 제공하는 보안 네트워크 기술입니다.  

### II. VPN의 구현 방식 및 서비스 형태 비교

**1. 서비스 형태(토폴로지)에 따른 비교**  

* **Intranet VPN:** 본사와 지사 간의 고정된 네트워크를 연결(Site-to-Site). 주로 IPSec 기반의 전용 장비 사용.  
* **Extranet VPN:** 기업과 협력업체 간의 네트워크 연결. 보안 정책 관리가 중요함.  
* **Remote Access VPN:** 출장자, 재택근무자 등 이동 사용자가 본사 네트워크에 접속(Client-to-Site). 주로 SSL VPN 활용.  

**2. OSI 7 Layer 구현 계층에 따른 비교**  

| 계층 | 대표 프로토콜 | 특징 | 주요 용도 |
| --- | --- | --- | --- |
| **L2 (데이터링크)** | L2TP, PPTP | 원격 사용자의 L2 접속 제공, 보안성이 상대적으로 낮음 | 초기 원격 접속 |
| **L3 (네트워크)** | IPSec | 네트워크 계층 암호화(AH, ESP), 호환성 우수, 설정 복잡 | 본-지사 간 연결 (Site-to-Site) |
| **L4+ (세션/응용)** | SSL/TLS | 웹 브라우저 기반 암호화, 별도 클라이언트 S/W 불필요 | 원격 접속 (Remote Access) |

### III. SSL VPN의 동작 원리 및 특징

**1. SSL VPN의 개념**  
웹 브라우저에 기본 탑재된 SSL/TLS 프로토콜을 활용하여, 별도의 전용 소프트웨어 설치 없이 언제 어디서나 안전하게 내부 네트워크 애플리케이션에 접근할 수 있는 VPN 기술입니다.  

**2. 핵심 구성 기술**  

* **Authentication (인증):** 접속자의 신원 확인 (ID/PW, OTP, PKI 기반 인증서 등)  
* **Tunneling & Encryption (암호화):** 대칭키(데이터 암호화)와 비대칭키(세션키 교환)의 혼합 사용  
* **Access Control (접근 제어):** 사용자 그룹별로 접근 가능한 애플리케이션 리소스 제한  

**3. IPSec VPN 대비 SSL VPN의 장점**  

* **도입 및 관리 용이성:** 클라이언트 설치가 필요 없어 배포 및 유지보수 비용 절감  
* **세밀한 접근 제어:** 네트워크 전체가 아닌 특정 애플리케이션 단위의 권한 제어 가능  
* **포트 유연성:** 기본 HTTPS 포트(443)를 사용하여 방화벽 통과가 자유로움

## 2. 가상화 기술: 가상머신(VM)과 컨테이너 비교

### I. 클라우드 인프라의 핵심, 가상화 기술의 개요

물리적 IT 리소스(서버, 스토리지, 네트워크)를 논리적인 리소스로 추상화하여, 하드웨어 종속성을 제거하고 자원 활용률과 민첩성을 극대화하는 클라우드 컴퓨팅의 기반 기술입니다.  

### II. 가상머신(VM)과 컨테이너의 아키텍처 비교

**1. 가상머신 (Virtual Machine / Hypervisor 기반)**  

* Host OS 위에 Hypervisor(VMware, KVM 등)가 위치하며, 각각의 VM마다 독립적인 Guest OS를 구동하는 방식입니다.  
* **특징:** 완벽한 격리성(Isolation)을 제공하지만, Guest OS 부팅 및 커널 구동으로 인해 무겁고 리소스 오버헤드가 큽니다.  

**2. 컨테이너 (Container / OS-Level Virtualization)**  

* Host OS의 커널을 공유하면서 프로세스, 네트워크, 마운트 공간 등을 논리적으로 격리(Namespace, Cgroups 등)하여 애플리케이션을 구동하는 방식입니다.  
* **특징:** Guest OS가 없어 매우 가볍고, 밀집도가 높으며, 수 초 내에 구동되어 마이크로서비스 아키텍처(MSA)에 적합합니다.  

### III. 가상머신과 컨테이너 상세 비교

| 구분 | 가상머신 (VM) | 컨테이너 (Container) |
| --- | --- | --- |
| **가상화 수준** | 하드웨어 레벨 (Hypervisor) | OS 커널 레벨 (Docker 엔진 등) |
| **운영체제(OS)** | 독립된 Guest OS 필요 | Host OS 커널 공유 |
| **구동 속도 / 크기** | 분(Minute) 단위 / GB 단위 | 초(Second) 단위 / MB 단위 |
| **격리성 / 보안** | 강력함 (하드웨어 수준 격리) | 상대적 취약함 (커널 공유) |
| **적합한 워크로드** | 레거시 앱, 강력한 보안이 필요한 환경 | MSA, CI/CD, 클라우드 네이티브 앱 |

## 3. 데이터베이스 파티셔닝(Partitioning)

### I. 대용량 데이터베이스 파티셔닝의 개요

데이터베이스의 테이블이나 인덱스를 논리적으로는 하나의 객체로 유지하되, 물리적으로는 여러 개의 작은 단위(Partition)로 분할하여 분산 저장하고 관리하는 기법입니다.  

### II. 파티셔닝 추진 목적

* **성능 향상 (Performance):** 특정 파티션만 스캔(Partition Pruning)하여 쿼리 응답 속도 최적화  
* **가용성 확보 (Availability):** 특정 파티션에 장애가 발생해도 다른 파티션은 정상 서비스 가능  
* **관리 용이성 (Manageability):** 백업, 복구, 데이터 삭제(Drop Partition) 작업을 파티션 단위로 빠르게 수행  

### III. 파티셔닝의 종류

* **수평 분할 (Horizontal Partitioning):** 테이블의 '행(Row)'을 기준으로 분할. 샤딩(Sharding)과 유사한 개념.  
* **수직 분할 (Vertical Partitioning):** 테이블의 '열(Column)'을 기준으로 분할. 자주 쓰이는 컬럼과 그렇지 않은 컬럼을 분리하여 I/O 감소.  

### IV. 수평 파티셔닝의 분할 기준 (기법)

| 분할 기준 | 설명 | 장점 / 적용 사례 |
| --- | --- | --- |
| **Range (범위)** | 연속적인 숫자나 날짜 범위 기준 | 과거 데이터 관리(이력 테이블), 특정 기간 조회 |
| **Hash (해시)** | 해시 함수 결과값을 기준으로 분산 | 데이터 편향 방지, 균등한 부하 분산 처리 |
| **List (목록)** | 특정한 값(코드, 지역 등) 목록 기준 | 카테고리별 명확한 분할 (예: 부서코드, 국가코드) |
| **Composite (복합)** | 두 개 이상의 기법을 조합 (예: Range-Hash) | 단일 파티셔닝의 한계 보완, 대규모 DB에 필수 |

## 4. SDN 아키텍처와 OpenFlow

### I. SDN(Software Defined Networking)의 개요

네트워크 장비에서 제어부(Control Plane)와 전송부(Data Plane)를 논리적으로 분리하고, 중앙의 소프트웨어 컨트롤러를 통해 네트워크를 프로그래밍 가능하게 만드는 차세대 네트워크 구조입니다.  

### II. SDN의 아키텍처 구조

SDN은 크게 3개의 계층(Layer)과 이를 연결하는 2개의 인터페이스(API)로 구성됩니다.  

* **Application Layer:** 방화벽, 로드밸런서, 트래픽 엔지니어링 등 네트워크 응용 프로그램  
* **Control Layer (SDN Controller):** 네트워크 전체를 조망(Topology)하고 제어하는 두뇌 역할  
* **Infrastructure Layer (Data Plane):** 단순화된 네트워크 스위치들로, 컨트롤러의 명령에 따라 패킷 포워딩만 수행  
* **Northbound API:** Application 계층과 Control 계층 간 통신 (주로 REST API 사용)  
* **Southbound API:** Control 계층과 Infrastructure 계층 간 통신 (대표적으로 **OpenFlow** 프로토콜 사용)  

### III. OpenFlow 프로토콜의 메커니즘

OpenFlow는 SDN의 Southbound API 표준 제정을 주도한 프로토콜로, 컨트롤러가 스위치의 플로우 테이블(Flow Table)을 제어할 수 있게 합니다.  

* **플로우 테이블 (Flow Table):** 패킷의 매칭 조건(Match Fields), 처리 명령(Action), 통계 정보(Counters)로 구성. 스위치는 유입된 패킷을 이 테이블에 대조하여 처리.  
* **동작 원리:**  
1\. 스위치에 새로운 패킷 도착 시 플로우 테이블 검색.  
2\. 매칭되는 규칙이 없으면(Table Miss) 컨트롤러로 패킷 헤더 전송(Packet-In).  
3\. 컨트롤러가 라우팅 계산 후 새로운 규칙을 스위치에 하달(Packet-Out / Flow-Mod).  
4\. 스위치는 수신된 규칙에 따라 패킷을 포워딩.

## 5. 빅데이터 시스템 구축 및 활용을 위한 기술

### I. 빅데이터 시스템 아키텍처의 개요

빅데이터 생태계는 3V(Volume, Velocity, Variety) 특성을 갖는 데이터를 수집, 저장, 처리하고, 이를 통해 비즈니스 인사이트를 도출(분석 및 시각화)하는 일련의 데이터 파이프라인으로 구성됩니다.  

### II. 시스템 구축을 위한 요소 기술

* **데이터 수집 (Collection):**  
* 정형 데이터: Sqoop (RDBMS 데이터 추출)  
* 비정형/로그: Flume (대용량 로그 수집), Logstash  
* 실시간 스트리밍: Apache Kafka (고성능 분산 메시징 큐)  


* **데이터 저장 (Storage):**  
* 분산 파일 시스템: HDFS (Hadoop Distributed File System)  
* 클라우드 스토리지: Amazon S3 (데이터 레이크 구축의 핵심)  
* NoSQL: HBase, Cassandra, MongoDB (스키마리스, 빠른 읽기/쓰기)  


* **데이터 처리 (Processing):**  
* 배치 처리: MapReduce, Apache Spark (인메모리 기반으로 MR 대비 속도 획기적 향상)  
* 스트림 처리: Spark Streaming, Apache Flink  



### III. 시스템 활용을 위한 요소 기술

* **데이터 분석 및 질의 (Analytics):** Hive (SQL-like 질의어 지원), Presto/Trino (대화형 빠른 쿼리)  
* **기계학습 및 AI (ML):** Spark MLlib, TensorFlow/PyTorch 연동을 통한 예측 모델링  
* **시각화 (Visualization):** Tableau, Power BI, Apache Superset, Kibana (데이터의 직관적 해석)  
* **데이터 거버넌스 및 카탈로그:** Apache Atlas (데이터 계보 추적 및 메타데이터 관리)

## 6. 딥러닝: Parameter vs Hyperparameter 및 튜닝 기법

### I. 딥러닝 모델 파라미터의 이해

딥러닝 모델의 성능은 입력 데이터가 출력으로 변환되는 과정에서 연산에 사용되는 다양한 변수들을 어떻게 최적화하느냐에 달려 있습니다. 이 변수들은 모델이 스스로 학습하는 값(Parameter)과 사람이 직접 설정해야 하는 값(Hyperparameter)으로 나뉩니다.  

### II. Parameter와 Hyperparameter 비교

| 구분 | 파라미터 (Parameter) | 하이퍼파라미터 (Hyperparameter) |
| --- | --- | --- |
| **정의** | 모델 내부에 존재하며 데이터로부터 **스스로 학습**되는 가중치 변수 | 모델 외부에서 **사용자(엔지니어)가 직접 설정**해 주는 제어 변수 |
| **결정 주체** | 최적화 알고리즘 (Gradient Descent 등) | 연구자 / 엔지니어 / 튜닝 알고리즘 |
| **예시** | 가중치(Weights), 편향(Bias) | 학습률(Learning Rate), 배치 크기(Batch Size), 은닉층/노드 수, 에포크(Epoch) |

### III. 하이퍼파라미터 튜닝 방법론

1\. **Grid Search:** 설정할 값들의 모든 가능한 조합을 격자 형태로 만들어 교차 검증을 통해 최적값을 찾는 방식. 직관적이나 리소스 소모가 극심함.  
2\. **Random Search:** 지정된 범위 내에서 임의의 조합을 추출하여 테스트하는 방식. 불필요한 탐색을 줄여 Grid Search보다 효율적으로 최적값을 찾을 확률이 높음.  
3\. **Bayesian Optimization (베이지안 최적화):** 이전 탐색 결과를 바탕으로 사전 지식(Prior)을 업데이트하며 다음 탐색 후보를 지능적으로 선택하는 방식 (Gaussian Process 활용).  
4\. **AutoML (Automated Machine Learning):** 데이터 전처리부터 알고리즘 선택, 하이퍼파라미터 튜닝까지 머신러닝의 전 과정을 자동화하는 최신 기법.

## 7. 레그테크(RegTech)

### I. 레그테크의 정의와 필요성

* **정의:** 규제(Regulation)와 기술(Technology)의 합성어로, IT 기술(AI, 빅데이터, 클라우드 등)을 활용하여 금융 규제 준수(Compliance) 업무를 자동화하고 효율화하는 기술입니다.  
* **필요성:** 핀테크 발전과 글로벌 규제(바젤 III, GDPR, 자금세탁방지 등) 강화로 인해 금융권의 컴플라이언스 비용과 벌금 리스크가 폭증함에 따라, 이를 선제적이고 저비용으로 해결할 대안이 필요해졌습니다.  

### II. 레그테크의 주요 특징

* **Agility (민첩성):** 얽혀있는 데이터 세트를 분리 및 조직화하여 규제 변화에 즉각 대응  
* **Speed (속도):** 수작업 기반의 보고서를 자동화하여 실시간 분석 및 실시간 규제 보고 가능  
* **Integration (통합):** 표준 API를 통해 기존 금융 시스템 및 다수 기관의 데이터와 단일 솔루션으로 통합  
* **Analytics (분석력):** 빅데이터 분석 및 머신러닝을 활용한 패턴 인식을 통해 사기 방지(FDS) 및 위험 사전 예측  

### III. 추진 동향 및 도입 시 고려사항

* **추진 동향:** 글로벌 금융기관들은 KYC(고객확인), AML(자금세탁방지), 이상거래탐지(FDS) 분야에 AI 기반 레그테크를 적극 도입 중이며, 감독기관 역시 섭테크(SupTech, 감독과 기술의 융합)로 진화하고 있습니다.  
* **도입 시 고려사항:**  
* **규제 표준화 API 구축:** 레거시 시스템과 레그테크 솔루션 간의 원활한 연동을 위한 오픈 API 표준화 필요.  
* **데이터 프라이버시 보호:** 개인정보를 대량으로 분석하므로 비식별화 및 개인정보보호법 준수 방안 마련.  
* **설명 가능한 AI (XAI) 적용:** 인공지능이 내린 규제 위반 판정에 대해 감독기관과 고객에게 논리적인 근거를 제시할 수 있어야 함.

## 8. 머신러닝 앙상블(Ensemble) 기법

### I. 앙상블 기법의 개요

단일 머신러닝 모델(Weak Learner)의 한계를 극복하기 위해 여러 개의 모델을 생성하고, 이들의 예측 결과를 결합함으로써 단일 모델보다 더 높은 신뢰성과 강력한 예측 성능(Strong Learner)을 얻는 기법입니다. (집단 지성의 원리)  

### II. 주요 앙상블 기법의 원리와 특징

**1. 배깅 (Bagging, Bootstrap Aggregating)**  

* **원리:** 원본 데이터에서 복원 추출(Bootstrap)을 통해 여러 개의 데이터 샘플셋을 만들고, 각각 독립적으로 모델을 학습시킨 후 결과를 투표(Voting)나 평균(Averaging)으로 결합합니다.  
* **특징:** 과적합(Overfitting) 방지 및 분산(Variance) 감소에 탁월함. 서로 독립적이라 병렬 처리가 가능함.  
* **대표 알고리즘:** Random Forest (Decision Tree의 앙상블)  

**2. 부스팅 (Boosting)**  

* **원리:** 여러 모델을 순차적으로 학습시킵니다. 이전 모델이 잘못 예측한(오답) 데이터에 가중치(Weight)를 부여하여, 다음 모델이 그 오류를 보완하도록 집중적으로 학습하는 방식입니다.  
* **특징:** 편향(Bias) 감소와 정확도 향상에 뛰어나지만, 순차 학습이므로 속도가 상대적으로 느리고 과적합 위험이 있습니다.  
* **대표 알고리즘:** AdaBoost, Gradient Boosting Machine(GBM), XGBoost, LightGBM  

**3. 스태킹 (Stacking)**  

* **원리:** 서로 다른 종류의 알고리즘(예: SVM, Random Forest, KNN 등)을 사용하여 1차 예측값들을 생성하고, 이 예측값들을 다시 입력 데이터(피처)로 삼아 최종 메타 모델(Meta Model)을 학습시키는 방식입니다.  
* **특징:** 캐글(Kaggle) 등 데이터 분석 경진대회에서 성능을 극한으로 끌어올릴 때 주로 사용됩니다.
