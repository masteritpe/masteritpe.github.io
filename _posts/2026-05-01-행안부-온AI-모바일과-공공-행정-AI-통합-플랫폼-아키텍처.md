## 1. 뉴스 요약

시사일보·다음·정보통신신문·지디넷코리아 등 보도에 따르면, **행정안전부가 「온AI(On-AI)」 모바일 서비스를 개시**하면서 **중앙부처 공무원이 네이버·삼성SDS의 AI 협업 도구를 본격 사용**하기 시작했다. 핵심은 ① **PC 기반 「온나라 2.0」 환경의 AI 기능을 모바일까지 확장**하여 보고서 초안·문서 검색·일정 관리·민원 응대를 단말 단에서 처리하고, ② **민간 기업의 LLM 협업 툴(네이버 클로바·삼성SDS Brity AI 등)을 정부 표준 환경에 포팅**해 부처 공통으로 사용하며, ③ 같은 흐름에서 정부가 **「공무원 바이브코딩(Vibe Coding)」 AI 에이전트 워크숍**을 열어 비개발자 공무원이 직접 자동화 에이전트를 만드는 시민 개발자(Citizen Developer) 역량을 확산한다는 것이다. 배경에는 **국정 디지털전환(DX)·공공 생산성 둔화·민원 폭증**이라는 압력이 있고, 부처별 분산된 AI 도입이 **사일로(Silo)**·**중복투자**·**보안 분열**을 낳아 왔다는 문제의식이 있다. 이번 정책은 **공공 행정 AI를 「부처별 도입」에서 「정부 공통 플랫폼·표준 SDLC」로 전환**하려는 시도이며, 소프트웨어공학적으로는 **MSA·BFF·Zero-Trust·MLOps**가 결합한 GovTech 플랫폼 아키텍처 과제를 던진다.

## 2. 핵심 개념

### 공공 행정 AI 통합 플랫폼 계층 구조

```
┌──────────────────────────────────────────────────────────┐
│ 채널 계층  : 모바일(온AI) / 웹(온나라2.0) / 챗봇 / 음성     │
├──────────────────────────────────────────────────────────┤
│ BFF/API 계층 : 부처별 BFF + 정부 공통 API Gateway          │
├──────────────────────────────────────────────────────────┤
│ AI 서비스 계층: 문서요약·검색·번역·코드생성(Agent)            │
├──────────────────────────────────────────────────────────┤
│ 모델 계층  : 정부 sLLM·민간 LLM(클로바·Brity 등) + RAG     │
├──────────────────────────────────────────────────────────┤
│ 데이터 계층 : 부처 문서·법령·민원 + 권한·등급 분리           │
├──────────────────────────────────────────────────────────┤
│ 인프라 계층 : G-클라우드·CSAP / 인증·로깅·관측              │
└──────────────────────────────────────────────────────────┘
```

### 부처별 AI 도입 vs 공공 AI 통합 플랫폼

다음 표는 두 모델의 차이를 정리한다.

| 구분 | 부처별 개별 도입 | 공공 AI 통합 플랫폼 |
|------|------------------|---------------------|
| 거버넌스 | 부처별 분산 의사결정 | 행안부·과기정통부 공통 표준 |
| 모델 운영 | 부처별 별도 LLM·라이선스 | 정부 sLLM + 민간 멀티 LLM 라우팅 |
| 데이터 | 부처 사일로 | 메타데이터·권한 기반 공유 |
| 보안 | 부처 ISMS-P 별도 인증 | CSAP·Zero-Trust 공통 |
| 비용 | 중복투자 | 단가 협상·재사용 |
| 개발 | 외주 SI 중심 | 시민개발자(노코드)+SI 혼합 |
| 운영 | 수기 모니터링 | MLOps·옵저버빌리티 표준 |

### 시민 개발자(Citizen Developer)와 바이브코딩

- **정의**: 본업이 개발이 아닌 공무원이 LLM·노코드 도구로 자동화 에이전트·앱을 직접 제작
- **도구**: 정부 표준 LLM 채팅, 노코드/로코드 RPA, 워크플로 빌더, 개발자 코파일럿
- **거버넌스**: 결과물 등록·심사, 보안 등급별 사용 범위 제한, 프롬프트·데이터 로깅
- **위험**: 섀도 IT, 환각(Hallucination)으로 인한 정책 오류, 개인정보 유출
- **통제**: 「가드레일(Guardrails) 라이브러리」, 정책 RAG, 결재 단계 인간 검토 필수화

### 공공 AI SDLC(소프트웨어 개발 수명주기)

```
요구분석 → 데이터·모델 선정 → 프롬프트·RAG 설계
   → 가드레일·평가셋 구축 → 보안·CSAP 검토
   → 시범운영(샌드박스) → 정식배포 → 모니터링·재학습
```

### 도입 시 고려사항

- **모델 라우팅**: 비용·성능·민감도 기반 자동 라우팅(소형 sLLM ↔ 대형 LLM)
- **검색 증강(RAG)**: 법령·내부규정·전결 기준 인덱싱, 최신성 보장
- **데이터 분류**: 공개·내부·민감 등급별 분리 학습·분리 추론
- **감사 추적**: 모든 프롬프트·응답·인용 출처 로그 보존(개정 행정조사기본법 대응)
- **사용자 경험(UX)**: 모바일 우선·접근성(WCAG)·다국어
- **연속 평가**: 모델 드리프트·환각률·민원 응대 정확도 정기 측정

## 3. 기술사 출제 포인트

- **공공 AI 통합 플랫폼의 정의·필요성**: 부처별 사일로·중복투자 해소
- **부처별 도입 vs 통합 플랫폼 비교**: 거버넌스·보안·비용·운영
- **공공 AI 6계층 아키텍처와 계층별 책임**: 채널~인프라
- **BFF(Backend for Frontend) 패턴의 적용**: 모바일·웹 채널 분리
- **민간 LLM과 정부 sLLM의 멀티 모델 라우팅 전략**
- **시민 개발자(Citizen Developer)·바이브코딩 도입 시 통제 방안**
- **정부 SDLC와 민간 SDLC 차이**: CSAP·감사·개인정보·전자정부 표준
- **공공 AI 가드레일 라이브러리의 구성 요소**: 프롬프트 인젝션 방어, 출력 검열
- **공공 AI MLOps 운영 지표**: 응답 정확도·환각률·인용 적합성·SLA
