## 1. 뉴스 요약

글로벌 데이터 레이크 분석 시장이 연 25% 성장률을 기록하며 데이터 기반 의사결정 경쟁이 본격화되고 있다. 기업들은 기존 데이터 웨어하우스와 데이터 레이크의 한계를 넘어, 두 아키텍처의 장점을 결합한 '데이터 레이크하우스(Data Lakehouse)' 도입을 가속화하고 있다. 국내에서도 AI 기반 빅데이터 분석 플랫폼이 속속 출시되며, 비정형 데이터와 정형 데이터를 통합 분석하는 수요가 급증하고 있다.

## 2. 핵심 개념

### 데이터 아키텍처 진화 경로
- **데이터 웨어하우스(DW)**: 정형 데이터 중심, 스키마 온 라이트(Schema-on-Write), 높은 쿼리 성능
- **데이터 레이크(Data Lake)**: 비정형/반정형 포함, 스키마 온 리드(Schema-on-Read), 유연한 저장
- **데이터 레이크하우스**: DW의 트랜잭션 관리 + 레이크의 유연성을 결합한 통합 아키텍처

### 데이터 레이크하우스 핵심 구성요소

| 구성요소 | 역할 | 대표 기술 |
|----------|------|-----------|
| 오픈 테이블 포맷 | ACID 트랜잭션, 스키마 진화, 타임트래블 지원 | Delta Lake, Apache Iceberg, Apache Hudi |
| 메타데이터 레이어 | 데이터 카탈로그, 스키마 관리, 접근 제어 | Unity Catalog, AWS Glue, Hive Metastore |
| 쿼리 엔진 | SQL 분석, ML 워크로드 실행 | Spark SQL, Trino, Dremio |
| 스토리지 레이어 | 저비용 오브젝트 스토리지 활용 | S3, ADLS, GCS, MinIO |

### 데이터 웨어하우스 vs 레이크 vs 레이크하우스 비교

| 항목 | 데이터 웨어하우스 | 데이터 레이크 | 데이터 레이크하우스 |
|------|-------------------|---------------|---------------------|
| 데이터 유형 | 정형 | 정형/반정형/비정형 | 정형/반정형/비정형 |
| 스키마 | Schema-on-Write | Schema-on-Read | 양방향 지원 |
| ACID 트랜잭션 | 지원 | 미지원 | 지원 |
| 비용 | 높음 (전용 스토리지) | 낮음 (오브젝트 스토리지) | 낮음 (오브젝트 스토리지) |
| 데이터 거버넌스 | 강함 | 약함 (데이터 스왐프 리스크) | 강함 |
| ML/AI 워크로드 | 제한적 | 네이티브 지원 | 네이티브 지원 |

## 3. 기술사 출제 포인트

### 예상 문제
1. **데이터 레이크하우스(Data Lakehouse)의 개념과 기존 데이터 웨어하우스, 데이터 레이크와의 차이점을 비교 설명하시오**
2. **오픈 테이블 포맷(Delta Lake, Apache Iceberg, Hudi)의 아키텍처와 ACID 트랜잭션 구현 방식을 설명하시오**
3. **엔터프라이즈 데이터 플랫폼을 레이크하우스 아키텍처로 전환하기 위한 마이그레이션 전략을 수립하시오**

### 답안 구성 가이드
- DW → Data Lake → Lakehouse 진화 배경과 각 아키텍처의 한계점 설명
- 레이크하우스의 4대 핵심 특성: ACID 트랜잭션, 스키마 강제/진화, 통합 거버넌스, BI/ML 통합
- 오픈 테이블 포맷의 역할: 메타데이터 기반 트랜잭션 로그로 데이터 일관성 보장
- 실무 고려사항: 기존 ETL 파이프라인 전환, 데이터 품질 관리, 비용 최적화

## 4. 관련 토픽 연계

- **데이터 메시(Data Mesh)**: 도메인 중심 분산 데이터 아키텍처와 레이크하우스의 관계
- **데이터 거버넌스**: 통합 카탈로그, 데이터 리니지(Lineage), 접근 제어 체계
- **실시간 분석**: 스트리밍 데이터 처리(Kafka, Flink)와 레이크하우스 통합
- **MLOps**: Feature Store 연동, ML 파이프라인에서의 레이크하우스 활용
- **데이터 품질 관리**: 데이터 스왐프 방지를 위한 데이터 계약(Data Contract)과 품질 메트릭
