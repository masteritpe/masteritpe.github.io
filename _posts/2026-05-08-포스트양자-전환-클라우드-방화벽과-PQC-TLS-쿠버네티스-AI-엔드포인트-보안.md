## 1. 뉴스 요약

지티티코리아(2026-05-07) 보도에 따르면, **클라우드 방화벽** 영역에서 **포스트양자 암호(PQC) 전환을 앞두고** ① **TLS 1.3 기반 PQC 하이브리드 키교환 검증**, ② **쿠버네티스 워크로드 간 mTLS의 PQC 마이그레이션 검증**, ③ **AI 모델 추론 엔드포인트(Inference Endpoint)의 PQC 보호 검증**이 본격화되고 있다. 미국 NIST가 2024년 ML-KEM(Kyber)·ML-DSA(Dilithium)·SLH-DSA(SPHINCS+)·HQC를 표준화한 이후, 클라우드 사업자·CSP·보안 솔루션 업체들이 **PQC 하이브리드 모드(고전+PQC) 기본 활성화**를 진행 중이며, **컨테이너·서비스 메시·API 게이트웨이·WAF/방화벽·LLM 엔드포인트**까지 포괄하는 **암호 민첩성(Crypto-Agility) 검증**이 요구된다. 정보보안 관점에서 이 흐름은 ① **클라우드 네이티브 환경의 PQC 마이그레이션 로드맵**, ② **TLS 1.3 PQC 하이브리드 운용**, ③ **K8s 서비스메시 mTLS PQC 전환**, ④ **AI/LLM 엔드포인트 PQC 보호**, ⑤ **암호 민첩성 거버넌스**가 통합되어야 하는 차세대 클라우드 보안 과제다.

## 2. 핵심 개념

### 클라우드 PQC 전환 보안 아키텍처

```
┌──────────────────────────────────────────────────────────────┐
│ ① 인터넷 경계: TLS 1.3 + PQC 하이브리드(X25519+ML-KEM)        │
├──────────────────────────────────────────────────────────────┤
│ ② 클라우드 방화벽/WAF: PQC 인증서·검증·딥패킷 검사           │
├──────────────────────────────────────────────────────────────┤
│ ③ API 게이트웨이: PQC mTLS·서명 검증·키 회전                 │
├──────────────────────────────────────────────────────────────┤
│ ④ 서비스 메시(K8s): Istio/Linkerd mTLS의 PQC 인증서 발급      │
├──────────────────────────────────────────────────────────────┤
│ ⑤ 워크로드 ID: SPIFFE/SPIRE 기반 PQC 단기 인증서             │
├──────────────────────────────────────────────────────────────┤
│ ⑥ AI 엔드포인트: LLM 추론 API의 PQC mTLS·요청 서명           │
├──────────────────────────────────────────────────────────────┤
│ ⑦ KMS·HSM: PQC 키 저장·서명·하드웨어 보호                    │
├──────────────────────────────────────────────────────────────┤
│ ⑧ 거버넌스: 암호 민첩성 정책·SBOM·CBOM·재인증 체계           │
└──────────────────────────────────────────────────────────────┘
```

### 고전 암호 vs PQC 하이브리드 vs 순수 PQC

다음 표는 세 방식의 차이를 정리한다.

| 구분 | 고전 암호(RSA·ECC) | PQC 하이브리드 | 순수 PQC |
|------|---------------------|----------------|----------|
| 양자 위협 내성 | 없음(Shor 공격 취약) | 한쪽 위반 시 다른 쪽 보호 | 양자 안전 |
| 호환성 | 가장 높음 | 양쪽 지원 필요 | PQC 지원 단말 한정 |
| 키·서명 크기 | 작음 | 중간(고전+PQC 합산) | 큼(서명 수 KB) |
| 성능 | 가장 빠름 | 다소 저하(추가 연산) | 알고리즘별 편차 |
| 표준화 | RFC 안정 | TLS 1.3 하이브리드 채택 진행 | NIST 2024 표준 |
| 적합 시점 | 단기 운영 | 전환 과도기(권장) | 장기 목표 |
| 운영 복잡도 | 낮음 | 중간 | 알고리즘별 차이 |
| 대표 사례 | RSA-2048·ECDHE | X25519+ML-KEM | ML-KEM·ML-DSA |

### 핵심 구성요소

- **PQC 표준 알고리즘**: ML-KEM(Kyber, 키캡슐화)·ML-DSA(Dilithium, 서명)·SLH-DSA(SPHINCS+)·HQC
- **TLS 1.3 PQC 하이브리드**: 키교환을 X25519+ML-KEM 등으로 결합, 양자·고전 양쪽 안전
- **서비스 메시 mTLS**: Istio/Linkerd가 발급하는 단기 인증서를 PQC로 전환
- **워크로드 ID(SPIFFE/SPIRE)**: 컨테이너·VM에 발급되는 SVID를 PQC 서명으로 발급
- **API 게이트웨이·WAF**: PQC 인증서 검증, 요청 서명, 딥패킷 검사 호환성
- **AI/LLM 엔드포인트**: 모델 추론 API의 mTLS·요청 무결성 서명·재전송 방지
- **KMS·HSM**: PQC 키 저장·서명, FIPS 140-3 준수, 하드웨어 보호
- **CBOM(Cryptography Bill of Materials)**: 시스템 내 사용 중인 암호 알고리즘 인벤토리
- **암호 민첩성(Crypto-Agility)**: 알고리즘 교체·키 회전·인증서 재발급 자동화

### 도입 시 고려사항

- **NIST PQC 표준 준수**: FIPS 203(ML-KEM)·204(ML-DSA)·205(SLH-DSA)·206(HQC) 정렬
- **하이브리드 우선 전환**: 호환성·롤백을 위해 PQC 하이브리드 모드부터 단계 적용
- **인증서·키 생명주기**: 키 길이·서명 크기 증가 → MTU·MSS·HSM 용량 재산정
- **성능 영향 측정**: TLS 핸드셰이크·서비스메시 mTLS 지연·CPU 사용률 벤치마크
- **레거시 호환성**: PQC 미지원 단말·라이브러리 대응 정책, 점진 전환 로드맵
- **암호 인벤토리(CBOM)**: 시스템·라이브러리·SaaS의 암호 사용 식별, SBOM 연계
- **거버넌스**: 암호정책위원회·내부통제, ISMS-P·전자서명법·전자정부법 정합
- **AI 엔드포인트 보호**: LLM 프롬프트·응답 무결성·재전송 방지, 모델 가중치 보호

## 3. 기술사 출제 포인트

- 포스트양자 암호(PQC) 도입 필요성과 양자 위협(Shor·Grover) 모델
- NIST PQC 표준(FIPS 203/204/205/206) 알고리즘별 특징 비교
- TLS 1.3 PQC 하이브리드 키교환 동작 원리
- 쿠버네티스·서비스 메시(Istio/Linkerd)의 mTLS PQC 전환 전략
- SPIFFE/SPIRE 기반 워크로드 ID와 PQC 인증서 발급
- 클라우드 방화벽·WAF·API 게이트웨이의 PQC 호환성 검증
- AI/LLM 추론 엔드포인트의 mTLS·요청 서명 보호 방안
- 암호 민첩성(Crypto-Agility) 설계 원칙과 자동화 방안
- CBOM(Crypto BOM)·SBOM 기반 암호 인벤토리 관리
- 하이브리드 vs 순수 PQC 단계별 마이그레이션 전략
- HSM·KMS·FIPS 140-3 준수 PQC 키 관리
- 제로트러스트 아키텍처와 PQC mTLS 통합
