### [경영/컨설팅] STEEPLE 분석

**[정의]**

- 기업 경쟁력에 영향을 미치는 7가지 거시적 외부 환경 요인을 체계적으로 파악·분석하는 전략 프레임워크
- PESTEL + Ethical (윤리적)

**[핵심/키워드]**

- PESTEL을 확장한 모델로, 환경(Ecological)·윤리(Ethical) 요인을 추가하여 ESG 경영 흐름 반영

* 분석요소
S - Social(사회적) : 인구구조, 문화, 라이프스타일, 소비패턴, 교육수준 변화
T - Technological(기술적) : 기술혁신, R&D, 자동화, 디지털전환, 특허
E - Economical(경제적) : GDP, 금리, 환율, 인플레이션, 소비심리, 경기사이클
E - Ecological(생태학적) : 환경규제, 탄소중립, 기후변화, 친환경정책, ESG
P - Political(정치적) : 정부정책, 정치적 안정성, 무역규제, 세제정책
L - Legal(법적) : 법률체계, 규제준수(Compliance), 소비자보호법, 노동법
E - Ethical(윤리적) : 기업윤리, 사회적책임(CSR), 공정거래, 반부패 기준

* 거시(Macro) 환경분석(통제 불가능) : PEST, PESTEL, STEEP, STEEPLE
* 미시적(Micro) 환경분석 : 3C, 5-Forces

### [경영/컨설팅] 가치사슬(Value Chain)

**[정의]**

- 기업이 고객에게 가치를 제공하는 과정에서 부가가치 창출에 직·간접적으로 관련된 일련의 활동·기능·프로세스를 연계하여 분석하는 기법

**[핵심/키워드]**

- 조직 내부환경 분석 도구
- 마진(Margin) 최대화가 목적

* 주활동 (Primary Activity) - 직접 부가가치 창출
- Inbound Logistics(물류투입) : 원자재 수령·보관·재고관리
- Operations(운영: 제조·생산) : 투입물을 최종제품으로 변환
- Outbound Logistics(물류산출) : 완제품 보관·배송·유통
- Marketing & Sales(마케팅·영업) : 구매 촉진·광고·채널관리
- Service(서비스) : 제품가치 유지를 위한 A/S·설치·교육

* 지원활동 (Support Activity) - 간접 부가가치 창출
- Procurement(구매조달) : 투입물 구매
- Technology Development(기술개발) : R&D, 공정기술
- Human Resource Management(인적자원관리) : 채용·교육·평가
- Firm Infrastructure(기업 인프라) : 재무·법무·기획·회계

### [경영/컨설팅] IT 투자성과 분석방법론

**[정의]**

- 정보시스템·IT 서비스·디지털 기술에 대한 투자가 재무적·비재무적 가치 창출에 얼마나 기여했는지 체계적으로 평가하는 분석 체계

**[핵심/키워드]**

- IT 투자의 효율성·효과성을 객관적으로 측정하여 의사결정 지원 및 투자 우선순위 설정에 활용
- 정량적(재무) + 정성적(비재무) 관점을 통합하여 IT 투자의 종합 가치를 평가

* 이익편익 분석(편익비용 분석)
- IT-ROI : 투자 대비 수익률, ROI = (편익-비용)/비용 × 100
- NPV(순현재가치) : NPV > 0이면 투자가치 있음
- IRR(내부수익률) : IRR > 기준수익률이면 투자가치 있음
- PP(투자회수기간) : 투자비용 회수에 걸리는 기간

* 실물옵션평가(블이몬) - 불확실성 하의 유연한 의사결정 가치 평가
- 블랙-숄츠(Black-Scholes) : 프로젝트 가치변화에 따른 IT 투자옵션 가치변화 산식
- 이항옵션(Binomial Option) : 상승·하락 두 가지 경우만 고려
- 몬테카를로 시뮬레이션 : 모든 경로 예측, 확률 분포 기반

* 다층분석(재비기리투미) - 복합관점 종합평가
- 재무적 관점 : NPV, ROI, PP
- 비즈니스 관점 : 전략 연계성, 성과 기여도
- 기술적 관점 : 성능, 표준 준수, 운영 용이성
- 리스크 관점 : 발생 가능성 × 영향도
- 투자 대안 평가 : 유사 프로젝트 수행경험 활용
- MECE 기법 : 상호배제·전체포괄 원칙으로 KPI 도출

### [경영/컨설팅] KPI(Key Performance Indictor)

**[정의]**

- 조직의 전략 목표 달성 수준을 정량적·정성적으로 측정·관리하기 위한 핵심 성과 지표 체계

**[핵심/키워드]**

* 절차
미션·비전·성과목표 확인 > KPI 수집·후보 KPI 생성 > 1차 성과지표 선정 > KPI 조정·확정 > KPI 상세화 > KPI 가중치 부여

* KPI 개발의 7대 원칙 (대적인구측기비)
- 대표성, 적절성, 인과성, 구체성, 측정가능성, 기한성, 비교가능성

* 좋은 KPI 조건 (SMART)
- Specific(구체성) : 명확하고 구체적인 측정 대상 정의
- Measurable(측정가능) : 정량적 수치 또는 기준으로 표현
- Achievable(달성가능) : 현실적으로 달성 가능한 목표 수준
- Relevant(연관성) : 전략 목표와 직접 연관된 지표
- Time-bound(기한) : 측정 주기 및 목표 달성 시점 명시

- BSC(균형성과표)와 연계
* BSC 4대 관점별 KPI 예시
- 재무관점 : ROI, 매출성장률, 비용절감률
- 고객관점 : 고객만족도(CSAT), NPS, 시장점유율
- 내부프로세스 : 불량률, 사이클타임, 프로세스 효율성
- 학습·성장 : 직원역량 향상률, 혁신제안 건수

### [경영/컨설팅] STP 전략, 시장세분화

**[정의]**

- 성공적인 마케팅 전략 설계의 핵심 3단계 프레임워크
- 잠재시장을 고객 요구·특징에 따라 수요층별로 분할하고, 목표시장을 선정하여 최적 포지셔닝 전략을 수립하는 기법

**[핵심/키워드]**

S - Segmentation(시장세분화)
- 세분화 조건 : 측정가능성·접근가능성·충분한 규모·실행가능성
* 세분화 방법에 따른 분류
- 경쟁적 좌표에 의한 시장 세분화 : 시장점유율과 제품수명주기에 따라 세분화
→ (예시) 공격시장, 방어시장, 철수시장, 포기시장
- 개인의 집합으로서의 시장 세분화 : 개인 특성 기준으로 세분화        
→ (예시) 인구통계변수, 생활유형변수, 지리적 변수, 개인변수
- 구매와 사용주체에 의한 시장 세분화        
→ (예시) 어린이시장, 주부시장, 가계시장, 조직체 시장
- 구매 행동측면에 따른 시장 세분화 : 구매량, 구매능력, 구매혜택 등에 따른 세분화        
→ (예시) 고가시장, 저가시장

T - Targeting(목표시장 선정)
* 타깃시장 선정 전략 유형
- 차별화 전략 : 복수 세그먼트별 개별 전략
- 비차별화(무차별) 전략 : 전체시장을 하나로 보고 단일 마케팅 전략 적용
- 집중화 전략 : 단일 틈새시장(니치시장) 집중 공략

P - Positioning(포지셔닝)
* 포지셔닝 전략 유형
- 속성에 의한 포지셔닝
- 사용자에 의한 포지셔닝
- 사용 상황에 의한 포지셔닝
- 경쟁제품에 의한 포지셔닝 : 소비자 인식 속에 경쟁사 대비 차별화된 위치 선점

### [경영/컨설팅] 기술수용주기

**[정의]**

- 소비자 유형별 행동패턴을 제품수명주기 모델과 결합하여 신기술·제품의 시장 확산 과정을 설명하는 모델

**[핵심/키워드]**

* 소비자 유형 5단계 (혁선전후지)
① 혁신자(Innovators) : 기술 자체에 매력 느낌, 위험 감수
② 선각수용자(Early Adopters) : 비전가(visionary), 레퍼런스 역할
③ 전기다수(Early Majority : 실용주의자, 검증된 제품 선호
④ 후기다수(Late Majority) : 회의론자, 표준이 된 후 수용
⑤ 지각 수용자(Laggards) : 전통주의자, 마지막 수용

* 수용주기 3단계
- 초기시장 : 혁신자 + 선각수용자
- 주류시장 : 전기다수 + 후기다수
- 말기시장 : 지각 수용자

→ 초기시장과 주류시장 사이에 대단절(캐즘) 발생

* 캐즘(Chasm)
- 초기시장에서 주류시장으로 진입 시 발생하는 일시적 수요 단절 현상
- 선각수용자(비전가)와 전기다수(실용주의자) 간 가치관 차이에서 발생
- 극복전략 : 디데이 전략, Bowling Alley(니치시장,틈새시장(볼링핀 전략)), Tonado 집중 공략 → 주류시장 확대

### [경영/컨설팅] Matrix 분석기법 - 포트폴리오 분석기법

**[정의]**

- 각 사업단위가 속한 시장의 성장률과 상대적 시장점유율을 기준으로 사업 포트폴리오를 평가하는 분석 기법

**[핵심/키워드]**

* BCG Matrix (보스턴 컨설팅 그룹)
- 기준 : 시장성장률(세로) × 상대적 시장점유율(가로)
- Stars(별) : 고성장·고점유 → 투자 집중 (미래 Cash Cow)
- Question Marks(물음표) : 고성장·저점유 → 선별 투자 또는 철수
- Cash Cows(젖소) : 저성장·고점유 → 수익 창출, 타 사업 지원
- Dogs(개) : 저성장·저점유 → 철수·매각 고려

* GE-McKinsey Matrix
- 기준 : 산업매력도(세로) × 사업경쟁력(가로) → 3×3 9개 셀
- 산업매력도 : 시장규모·성장률·수익성·경쟁강도
- 사업경쟁력 : 시장점유율·브랜드강점·생산능력·기술력
- BCG보다 다차원적 평가 가능

* 포트폴리오 전략 방향
- 투자확대 / 유지 / 선별적 투자 / 수확 / 철수

### [경영/컨설팅] 가트너 하이프 사이클(Hype Cycle)

**[정의]**

- 미국 IT 연구기업 Gartner가 신기술의 성숙도를 5단계 사이클로 시각화한 모델
- 특정 시점 기술의 시장 기대치와 성숙도를 평가하기 위한 프레임워크

**[핵심/키워드]**

* 기술 사이클 5단계 (기부환계생)

① 기술 촉발기 (Technology Trigger)
- 새로운 기술 등장, 미디어 관심 집중, 초기 시장 형성
- POC(Proof of Concept) 수준의 제품·서비스 등장

② 부풀려진 기대의 정점 (Peak of Inflated Expectations)
- 기대감 폭발, 미디어 과장 보도, 투자·관심 급증
- 일부 성공사례 등장, 과도한 낙관론 형성

③ 환멸 단계 (Trough of Disillusionment)
- 기대와 현실 격차 노출, 실패 사례 증가, 투자 감소
- 기술 개발사 구조조정, 미디어 관심 급감

④ 계몽 단계 (Slope of Enlightenment)
- 기술 성숙, 실제 적용 사례 증가, 투자 회복
- 2·3세대 제품 출시, 수익화 모델 구체화

⑤ 생산성 안정 단계 (Plateau of Productivity)
- 기술 안정화, 광범위한 실용 활용, 생산성 향상
- 주류 시장 채택 완성

### [경영/컨설팅] PI

**[정의]**

- 기존의 업무처리 방식을 고객지향으로 근본적으로 바꾸어 경쟁우위의 변화 대응력을 확보하는 프로세스 혁신 경영기법
- IT를 활용하여 기업의 핵심 프로세스를 재설계·최적화하며, BPR과 달리 점진적·지속적 개선에 초점

**[핵심/키워드]**

* 추진 3요소
- 프로세스 : 전 조직원 참여, 업무절차 최적화
- 정보기술 : 비전과 전략에 의한 Top-Down 전개, IT 기반 자동화
- 조직 : 조직 재설계, 조직 능력 향상, 사용자 및 IT 전문가 참여

* 추진 단계
현황진단 > 미래모습 설계 > 혁신과제 도출 > 실행계획 수립 > 실행 및 모니터링

* BPR vs PI 비교
- BPR : Zero-Base에서 급진적 재설계, 혁신 속도 빠름, 저항 큼
- PI : 기존 프로세스 기반 점진적·지속적 개선, 실행 용이

### [경영/컨설팅] BPR(Business Process Reengineering)

**[정의]**

- 기업의 핵심 업무 프로세스를 Zero-Base에서 근본적으로 재설계하는 경영 혁신 기법
- 비용·품질·서비스·속도 등 핵심 성과지표의 획기적 향상을 목적으로 기존 관습과 규칙을 무시하고 혁신적으로 재설계

**[핵심/키워드]**

* 4가지 핵심 키워드
- 근본적(Fundamental) : 왜 하는가를 질문, 전제와 관습 타파
- 혁신적(Radical) : 점진적 개선이 아닌 근본적 변화
- 극적(Dramatic) : 10% 개선이 아닌 10배 이상의 성과 향상
- 프로세스(Process) : 개별 기능이 아닌 전체 업무흐름 재설계

* 구성요소 (사기프)
- 사람 : 조직구성·역할분담, 교육/훈련, 변화관리
- 기술 : 정보기술(IT), 업무기술, 프로젝트 관리기술
- 프로세스 : 업무절차 간소화, 표준화, WorkFlow

* 추진단계 (8단계)
현 프로세스 이해 > 고객 Needs 파악 > 목표 설정 > 혁신 프로세스 설계 > 실행과제 도출 > 과제 해결 > 실행 > 점검

### [경영/컨설팅] ISP(Information Strategy Planning)

### [경영/컨설팅] ISMP(Information Strategy Management Planning)

**[정의]**

- 특정 SW개발사업에 대한 상세분석과 제안요청서 마련
- 비지니스 및 정보기술에 대한 현황과 요구사항을 분석
- 기능점수 도출이 가능한 수준까지 기능적, 기술적, 비기능적 요건을 상세히 기술하며 구축 전략 및 이행계획을 수립하는 활동

**[핵심/키워드]**

* 방법론 체계(착방업구이)
1\. 프로젝트 착수 및 참여자 결정 : 경영진 지원조직 형성, 프로젝트 수행조직편성, 프로젝트 계획수립
2\. 정보시스템 방향성 수립 : 정보화전략 검토, 벤치마킹, 추진범위 및 방향정의
3\. 업무 및 정보기술 요건 분석 : 현황분석, 업무요건분석, 정보기술 요건분석, 기술서검토 
4\. 구조 요건 정의 : 정보시스템 아키텍쳐 정의, 요건간 이행 연관성 분석, 요건 기술서 작성
5\. 정보시스템 구축 이행방안 수립 : 구축사업 계획수립, 분리발주 가능성평가, 예산수립, RFP작성, 구축업체 선정 및 평가

### [경영/컨설팅] IT Governance

**[정의]**

- 기업의 전략과 목표에 부합하도록 IT와 관련된 자원·프로세스를 통제·관리하는 체계
- IT가 비즈니스 가치를 창출하고 리스크를 최소화하도록 IT 의사결정 권한과 책임 구조를 명확히 정의
- 이사회·경영진 수준의 IT 통제 및 방향 제시

**[핵심/키워드]**

- 전략적 연계(Strategic Alignment) : ISP, ITA/EA를 통한 IT-비즈니스 정렬

- ITIL : IT 서비스 관리 모범 사례(Best Practice)

* IT Governance 필요성
- IT의 전략적 중요성 증대, 기업 리스크 확대
- IT 투자 효율성·책임성 요구, 컴플라이언스 강화

### [경영/컨설팅] ISO38500

**[정의]**

- IT거버넌스 국제 표준
- 이사진이 IT가 기업에 효율적, 효과적, 책임성있게 활용될 수 있도록 조직의 IT활동의 통제를 평가, 지휘, 모니터링 하는데 활용되는 국제 표준(평지모, EDM)
- ISO/IEC 38500:2024 
→ 평가 + 지휘 + 모니터링 + 이해관계자 참여

**[핵심/키워드]**

* 구현을 위한 6원칙(책전획성준행)
책임, 전략, 획득, 성과, 준거, 행동

* ISO38500:2024 → 2015 핵심 6원칙에서 ISO 37000에 정의된 거버넌스 원칙과 정합성 강화, 11원칙으로 확장 
- 주요원칙 : 목적
- 기초원칙 : 가치 모델, 전략, 감독, 책임
- 지원원칙 : 이해관계자 참여, 리더십, 데이터와 의사결정, 위험 거버넌스, 사회적 책임, 지속가능성 성과

### [경영/컨설팅] COBIT

**[정의]**

- 효과적인 IT Governance 실현 및 IT 통제 수준 진단을 위한 통제 프레임워크 (ISACA 제정)
- IT 프로세스·통제활동·성과지표를 연계하여 IT 경영목표 달성 수준을 체계적으로 평가
- RiskIT와 ValIT를 통합한 COBIT 5.0은 5대 원칙 기반 전사적 IT 거버넌스 체계 제공

**[핵심/키워드]**

* 성숙도 평가 6단계 (부초반정관최)
0-부재 : IT 통제 프로세스 전혀 없음
1-초기 : 비공식적 ad hoc 수행
2-반복 : 개인 역량에 의존
3-정의 : 표준화·문서화된 프로세스
4-관리 : 측정·모니터링 가능
5-최적 : 지속적 개선·자동화

* COBIT 5.0 - 5대 원칙 (통이비인거)
1\. 통합적 프레임워크 : IT·비즈니스 목표 통합
2\. 이해관계자 가치제고 : 모든 이해관계자 요구 충족
3\. 비즈니스 초점 : 비즈니스 목표에서 IT 목표 도출
4\. 인에이블러 기반 : 원칙/정책, 프로세스, 조직구조, 문화/윤리, 정보, 서비스, 인력
5\. 거버넌스와 관리 분리

* COBIT 5 프로세스 구조 (이효리가준선물)

- 지속적 개선
- 가치

* SVC(Service Value Chain): 계참설획운개
- 계획, 참여, 설계 및 전환, 획득 및 구축, 운영,개선

### [경영/컨설팅] SLA(Service Level Agreement)

**[정의]**

- IT서비스를 제공하거나 제공받기 위해 상호협의한 서비스 수준의 합의서
- IT아웃소싱의 서비스 수준을 측정, 평가, 개선하는 IT서비스 수준의 성과관리 방식에 대한 계약서

**[핵심/키워드]**

* 구성요소
- 기본계약서
- SOW(Statement Of Work) 
- SLM(Service Level Metrics): 서비스 수준관리
- SLO(Service Level Objective): 서비스 목표수준
- SLM(Service Level Measurement): 서비스성과측정
- SLR(Service Level Report): 서비스수준 보고
- 보상체계

* 절차 : 준비 > 협약 및 측정 > 운영 및 평가

* 계약시 고려사항
- 고객사 : 범위명확화, 평가지표, 평가수준
- 제공사 : 현실성, 개선회의체구성, 시범운영기간

### [경영/컨설팅] ISO 22301

**[정의]**

- 기업의 비즈니스 연속성 관리 시스템(BCMS: Business Continuity Management System) 국제표준
- PDCA 사이클 기반으로 조직의 비즈니스 연속성을 체계적으로 관리

**[핵심/키워드]**

* PDCA 4단계
- Plan(계획수립) : BCM 방침·목표·BIA 수행·복구전략 수립
- Do(실행 및 운영) : BCP 구현·훈련·교육·인식 제고
- Check(모니터링 및 검토) : 성과 모니터링·내부 감사·경영검토
- Act(유지 및 개선) : 시정조치·예방조치·지속적 개선

* 주요 내용 (조리계지운성개)
- 조직상황 : BCM 관리자 역할, 내·외부 이슈 분석
- 리더십(지도층) : 경영진 책무, 업무 연속성 방침
- 계획수립(기획) : 목적 제시, BCP 계획 수립, 위험요소 식별
- 지원 : 자원관리, 수행능력 구축, 의사소통
- 운영(운용) : 운영계획수립·통제, 모의훈련, BIA 수행
- 성과관리(성과평가) : 감시·측정·분석·평가·감사
- 개선 : 부적합 시정조치, 지속적 개선

* 관련 표준
- ISO 22301 ↔ BS 25999(영국) ↔ NFPA 1600(미국)

### [경영/컨설팅] BCP

**[정의]**

- 각종 재해·장애·재난으로부터 비즈니스 연속성을 보장하기 위한 종합 체계
- 위기관리를 기반으로 재해복구·업무복구·재개·비상계획을 포함하는 전사적 사업 연속성 계획
- ISO 22301(BCMS) 기반으로 BIA를 통한 정량적 의사결정, 핵심 업무 우선순위 기반으로 수립

**[핵심/키워드]**

* 핵심 개념
- ISO 22301 : BCP의 국제 표준 (BCMS)
- BIA(Business Impact Analysis) : 복구우선순위 파악, MTD·RPO·RTO 도출
- DR(Disaster Recovery) : BCP의 한 구성요소 (IT 시스템 복구 중심)
- MTD(Maximum Tolerable Downtime) : 업무 중단 최대 허용 시간
- RPO(Recovery Point Objective) : 데이터 복구 목표 시점 (데이터 손실 허용 범위)
- RTO(Recovery Time Objective) : 서비스 복구 목표 시간

* BCP 프레임워크 구성요소
- DRS(재해복구 시스템) : 비상대비체계 유지, 복구계획 수립
- BR(Business Recovery) : 업무복구
- BIA : 복구우선순위·MTD·RPO·RTO 파악
- Backup : 백업 데이터 송수신 및 보관
- Planning : 업무중요도 분류 및 선정

* BCP 프로세스
위험분석 → 업무영향분석(BIA) → 복구전략 수립 → 계획 개발 → 테스트·검증 → 유지보수

### [경영/컨설팅] DRS(Disaster Recovery System)

**[정의]**

- 재해복구계획의 원활한 수행을 지원하기 위하여 평상시에 확보하여 두는 인적, 물적 자원 및 이들에 대한 지속적인 관리체계 또는 시스템

**[핵심/키워드]**

* 유형: 독자구축, 공동구축, 상호구축
* 절차
- 재해복구 전략수립 → 업무영향분석(BIA) → DRS 설계 → DRS 구축 
→ DRS 운영 및 모의훈련

* 복구수준(MHWC)
- Mirrored Site: 동시데이터 처리 및 운영. 즉시 대체 가능
- Hot Site: 데이터 실시간 이중화
- Warm Site: 일부 장비만 구비
- Cold Site: 기본시설만 확보

### [경영/컨설팅] DRaaS, 서비스형 재해복구(Disaster Recovery as a Service)

**[정의]**

- 재해 발생 시 전체 시스템과 서비스의 복구를 클라우드를 통해 제공하는 서비스

**[핵심/키워드]**

* 목적 : 시스템 장애나 재해 발생 시 업무 연속성 확보

- DRaaS :“업무 시스템 복구 중심”→ 업무 연속성 보장
- BaaS : “파일 복구 중심”→ 데이터 보호

* 서비스 모델
- 관리형: 전체 재해복구과정 아웃소싱 
- 지원형: 기업이 DR과정을 통제하면서 통합, 테스팅 담당 
- DIY형: 기업이 DR 전체 과정을 자체관리

### [경영/컨설팅] BaaS, 서비스형 백업(Backup as a Service)

**[정의]**

- 클라우드를 통해 데이터를 자동 백업하고 복구할 수 있는 서비스

**[핵심/키워드]**

* 목적 : 데이터 손실 방지 및 복원
* 대상 : 파일, 폴더, 데이터베이스 등의 데이터 중심

* 유형
- 전체백업: 보조 스토리지 위치에 정기적으로 데이터를 복사 
- 차등백업: 마지막 전체 백업 이후에 변경된 데이터만 복사
- 증분백업: 이전 백업 이후 변경된 데이터만 복사

### [경영/컨설팅] SCM(Supply Chain Management)

**[정의]**

- 공급자에서 최종 소비자에 이르는 제품의 공급체인과 관련된 모든 과정을 효율적으로 통합하여 기업의 수익증가와 고객서비스 개선을 위한 정보화된 경영혁신기법

**[핵심/키워드]**

* SCM 핵심 기능
- 수요예측 : 판매·수요 데이터 분석을 통한 재고 최적화
- 조달 관리 : 공급업체 선정·평가·계약 관리
- 재고 관리 : JIT(Just In Time), 안전재고 설정
- 물류 관리 : 운송·창고·배송 최적화
- 주문 관리 : 고객 주문처리·이행·추적

* 핵심 지표
- 주문 이행률, 납기준수율, 재고회전율, 공급자 리드타임
- 채찍효과(Bullwhip Effect) : 최종 소비자에서 공급자 방향으로 수요 변동 증폭 현상

* 목적
- 재고 최소화·납기 단축·고객 서비스 향상·비용 절감
- ERP 시스템과 연계하여 실시간 공급망 가시성 확보

### [경영/컨설팅] PLM(Product Lifecycle Management)

**[정의]**

- 제품 기획에서 설계, 제조, 출시, 유지보수, 폐기에 이르기까지 제품의 전체 라이프 사이클을 관리하는 체계

**[핵심/키워드]**

* 핵심요소 (부제비도프설변)
- 부품정보관리 : 부품 사양·공급자·재고 통합 관리
- 제품구성 정보관리 : 제품 아키텍처·구성요소 정의
- BOM(Bill of Materials) 정보관리 : 설계BOM·생산BOM·서비스BOM 관리
- 도면문서 및 기술문서 관리 : CAD 도면·사양서·표준 문서
- 프로젝트 관리 : 개발일정·마일스톤·리소스 관리
- 설계협업 Workflow : 부서 간 설계변경 승인 프로세스
- 설계 변경관리 : 변경 요청→영향분석→승인→실행

* 목표
- 제품 데이터·프로세스·인력·기술을 연결하여 제품개발 기간 단축, 품질 향상, 비용 절감
- ERP·CAD·MES와 연계하여 전사적 제품 정보 관리

### [경영/컨설팅] MES(Manufacturing Execution System)

**[정의]**

- 제품 주문부터 완제품 출하까지 생산 현장의 모든 과정을 실시간으로 모니터링, 제어, 문서화하여 최적화하는 생산관리시스템
- Real Time 통합 생산 관리시스템
- Bottom-Up 방식

**[핵심/키워드]**

* MES의 기능(계진목추)
- 계획관리 : 자원할당 및 상태, 작업 및 상세일정관리, 생산단위 분배, 문서제어
- 진행관리 : 데이터 집계 및 취득, 근로관리
- 목표관리 : 품질관리, 공정관리
- 추적관리 : 유지보수관리, 생산추적 및 이력, 실행분석

### [경영/컨설팅] PHM

**[정의]**

- 센서 데이터·진단 알고리즘·예측 모델을 활용하여 장비·시스템의 건전성(Health)을 실시간 모니터링하고 고장 가능성을 예측하여 최적 유지보수 전략을 수행하는 기술 및 관리 체계

**[핵심/키워드]**

* 구성요소
- Monitoring(모니터링) : 센서 기반 실시간 상태 데이터 수집
- Diagnostics(진단) : 현재 결함·이상 상태 식별 및 원인 분석
- Prognostics(예측) : 잔여 유효수명(RUL: Remaining Useful Life) 예측
- Decision Support(의사결정 지원) : 유지보수 시점·교체여부·리스크 완화 전략 수립
- Maintenance Execution(유지보수 실행) : 예측 기반 정비 수행

* 유지보수 유형 비교
- 사후(BM) : 고장 발생 후 수리 → 가동 중단 비용 높음
- 예방(PM) : 주기적 정비 → 불필요한 작업 수행
- 예측(PdM/PHM) : 실시간 상태 및 데이터 기반 최적 시점 정비 → 비용 최소화

- 사후 유지보수(Corrective)→예방 유지보수(Preventive)→예측 유지보수(Predictive) 진화의 최종 단계
- IoT·AI·빅데이터를 결합하여 제조·항공·에너지 등 산업 전반에 적용

* 핵심 기술
- 진동·온도·압력 센서, 신호처리, ML 이상탐지
- 디지털 트윈 연계, 클라우드 IoT 플랫폼

### [경영/컨설팅] IT Compliance

**[정의]**

- 기업의 투명성 강화 및 리스크 관리를 위해 정부·관련기관이 제시한 각종 규제법안을 IT 관점에서 시스템화하여 준수하는 의무 활동
- 정보시스템 사용자가 고객정보보호·자료보관·재무보고 공시 등 규정·지침을 준수하는 과정

- 준법감시부처 : 법규준수 점검절차·준법활동 수립·이행 점검

- Compliance Manual : 내부통제 정책·절차·위규 대처방법 기술

2) 주요 법안 및 규제

- SOX법(Sarbanes-Oxley Act) : 회계감사 투명성, CEO/CFO 서명 의무
- GDPR : EU 개인정보보호 규정
- 개인정보보호법(국내) : 개인정보 수집·이용·파기 규정
- K-SOX : 국내 내부회계관리제도

3) IT Compliance 지원시스템
- RDW(Risk Data Warehouse): 리스크 관리 DW
- COSO : 회계 재무 감사 전문조직
- HRD : IT 인적 자원 개발
- DW(Data Warehouse)
- 정보보호시스템

### [프로젝트관리] ISO 21500:2012

**[정의]**

- 조직이 프로젝트를 효율적으로 수행하기 위해 적용할 수 있는 프로젝트 관리의 개념·프로세스·원칙에 대한 지침(Guidance)을 제공하는 국제 표준 (초판 2012년, 영국표준협회 제안)
- PMBOK과 상호 호환되도록 설계되어 국제적 공통 PM 프레임워크 제공
- 5개 프로세스 그룹 × 10개 주제 그룹 구조

**[핵심/키워드]**

&lt;프로세스 그룹 5단계&gt; (착계실통종)
1) 착수(Initiating) : 프로젝트 승인·헌장 수립
2) 계획(Planning) : 목표 달성 계획 수립
3) 이행(Implementing) : 자원 투입·산출물 생성
4) 통제(Controlling) : 성과 모니터링·변경·리스크 관리
5) 종료(Closing) : 결과 인수·문서화·교훈 정리

&lt;7대 원칙&gt;
무결성, 투명성, 책임, 의사소통, 이해관계자 존중, 공정성, 지속적 개선

&lt;10대 주제 그룹&gt;

### [프로젝트관리] Scope Creep(범위 확장)Gold Plating(과잉 품질)

**[정의]**

* Scope Creep(범위 확장)
- 프로젝트 범위가 공식적인 변경 승인 절차 없이 비공식적으로 확대되는 현상
- 처음 합의된 목표·인도물·요구사항에 포함되지 않은 추가 작업이나 기능이 점진적으로 포함되는 상황

* Gold Plating
- 고객이나 이해관계자가 요청하지 않았음에도 팀이 자의적으로 “더 좋은 결과물”을 만들기 위해 범위를 초과하는 작업을 수행하는 행위
- 명확한 범위 정의+철저한 변경통제

**[핵심/키워드]**

* Scope Creep(범위 확장)
&lt; 발생 원인 &gt;

* Gold Plating 
&lt; 발생 원인 &gt;

&lt; 주요 영향 &gt;

### [프로젝트관리] 요구사항 수집 기법

**[정의]**

&lt; 데이터 수집 &gt; 
1\. 인터뷰 (Interview)
- 사전에 준비된 질문과 즉흥 질문을 통해 정보를 수집(1:1, 다:다) 
- 비밀정보 획득에 효과적
2\. 브레인스토밍 (Brainstorming)
- 여러 사람이 모여 자유롭게 아이디어를 제시하는 회의 기법
- 아이디어 생성 중심
3\. 포커스 그룹 (Focus Group)
- 사전 선정된 전문가·이해관계자 간의 상호 토론을 통해 의견 수집
- 상호 토론 중심
4\. 설문지 및 설문조사 (Questionnaires & Surveys)
- 다수의 응답자에게 빠르게 자료 수집 가능
- 정량적 데이터 확보
5\. 벤치마킹 (Benchmarking)
- 유사 조직의 프로세스·성과와 비교하여 모범사례 및 개선점 식별
- 모범사례 분석 중심

**[핵심/키워드]**

&lt; 데이터 분석 &gt;
1\. 문서 분석 (Document Analysis)
- 기존 문서(계획서, 계약서, 규정, 이슈기록 등)를 검토
- 과거자료 분석에 효과적

&lt; 의사결정 &gt;
2\. 투표 (Voting)
- 여러 대안 중 우선순위·선호도 결정에 사용
- 집단 의사결정 기법(만장일치·과반수·다수결·독재형 등) 
3\. 다기준 의사결정 분석 (Multicriteria Analysis)
- 리스크, 가치, 불확실성 등 평가기준을 수치화하여 대안 비교
- 정량 평가 기반

&lt; 데이터 표현 &gt; 
4\. 친화도 다이어그램 (Affinity Diagram)
- 다수의 아이디어를 주제별 그룹으로 분류하여 분석
- 아이디어 구조화
5\. 아이디어/마인드 매핑 (Idea/Mind Mapping)
- 브레인스토밍 결과를 시각적으로 통합하여 관계 표현
- 연관관계 분석

### [프로젝트관리] 요구사항 추적 매트릭스 (Requirements Traceability Matrix, RTM) = 요구사항 추적표

### [프로젝트관리] WBS(Work Breakdown Structure)

**[정의]**

- 프로젝트 목표를 달성하기 위해 수행해야 할 전체 작업 범위를 산출물 중심으로 계층 구조로 세분화한 체계도 

- WBS 작성주체 : 프로젝트 관리자를 포함한 프로젝트 팀
- WBS 완성시점 : 각 작업패키지에 통제단위를 설정하고, 작업패키지에 대한 고유한 식별코드를 지정할 때 

* WBS 구성요소: 작계통분식사 
- 작업단위(Work Package): 최하위 작업 
- 계획단위(Planning Package): 일정 미확정
- 통제단위(Control Account): 성과측정을 위한 관리 통제 점 
- 분할(Decomposition): 세분화 기법 
- 식별코드(Code of Account): 작업패키지 식별자 
- WBS 사전: 각 구성요소에 대한 상세한 정보

**[핵심/키워드]**

&lt; WBS 작성 원칙 &gt; 
- 분할: 상위 산출물을 더 작고 관리 가능한 단위(Work Package)로 세분화하는 기법 

- 분할수준: 3~5레벨(Level) 정도가 표준

- Work Package 크기: 일정·비용·성과 측정이 가능한 최소 단위, 80시간 규칙(80-hour rule) 적용 

- 100% Rule: MECE하게 프로젝트 범위의 100%를 충족

- 연동계획(Rolling Wave Planning) : 가까운 시기의 작업은 상세하게, 먼 미래의 작업은 개략적으로 계획 후, 시점이 다가오면 세부 분해

### [프로젝트관리] 점진적 상세화(Rolling Wave Planning, 연동기획)

**[정의]**

- 프로젝트 초기에는 가까운 작업을 상세하게 계획하고, 먼 미래의 작업은 개략적으로 계획한 후 시점이 다가올수록 세부 분해하는 반복적·점진적 계획 수립 기법
- 불확실성이 높은 프로젝트에서 현실적인 계획 수립을 위해 WBS 최하위 단위를 단계적으로 구체화
- 애자일 방법론의 스프린트 계획과 유사한 개념으로 변화 대응력 확보

**[핵심/키워드]**

* Rolling Wave Planning 핵심 원칙
- 근시안적 상세계획 + 원거리 개략계획의 병행
- Work Package : 가까운 인도 시점 → 상세하게 분해
- Planning Package : 먼 인도 시점 → 개략적 수준 유지

* 적용 시점
- WBS 작성 시 전체 범위를 한번에 상세화하기 어려운 경우
- 프로젝트 진행에 따라 요구사항이 구체화되는 경우
- 대형·복잡·불확실성이 높은 프로젝트

* 관련 개념
- WBS 사전(Dictionary) : 각 Work Package에 대한 상세 정의
- Progressive Elaboration : 정보가 명확해질수록 계획을 구체화
- 연동기획(Rolling Wave) ↔ 전체기획(Complete Planning): 유연성 vs 명확성

* 애자일 연계
- 스프린트 백로그 : 현 스프린트는 상세, 미래 스프린트는 개략
- 제품 백로그 정제(Refinement) 활동과 동일한 개념

### [프로젝트관리] WBS작성과 활동정의 비교

**[정의]**

- WBS 작성(5.4)과 활동 정의(6.2)는 모두 프로젝트 분해 기법이나 관점이 상이
- WBS 작성: 인도물(Deliverable) 관점에서 Work Package 단위로 분해
- 활동 정의: 활동(Activity) 관점에서 Work Package를 세부 일정활동으로 분해

**[핵심/키워드]**

&lt;WBS 작성 vs 활동 정의 비교&gt;

구분          | 5.4 WBS 작성             | 6.2 활동 정의
산출물 관점 | 인도물(Deliverable) 중심  | 활동(Activity) 중심
분해 단위    | Work Package              | Activity (Schedule Activity)
도구         | WBS, WBS Dictionary     | 활동 목록, 활동 속성, 마일스톤 목록
목적         | 범위 정의·통제             | 일정 수립·통제의 기반
관리영역    | 5. 범위관리                  | 6. 일정관리
입력→출력  | 범위기술서→WBS           | WBS→활동 목록

* 핵심 차이
- WBS의 최하위 단위 = Work Package
- Work Package → (활동 정의) → 하나 이상의 Schedule Activity로 세분화
- Work Package는 일정계획 수립의 입력이 되며, Activity는 네트워크 다이어그램의 구성 요소

* 분해(Decomposition) 기법 : 두 프로세스 공통 활용

### [프로젝트관리] 프로젝트 일정 네트워크 다이어그램 (Project Schedule Network Diagram)

**[정의]**

- 프로젝트 활동 간의 논리적 순서(의존관계)를 도식화한 일정관리 도구 

- 각 활동의 선행·후행 관계, 흐름 방향, 병행 가능성 등을 시각적으로 표현
- 활동 간 의존관계·선후관계·병목구간을 식별하여 효율적 일정계획 수립

**[핵심/키워드]**

의존관계 속성 4가지(의임외내): 2가지 속성은 동시에 적용될 수 있음 
1) 의무적 의존관계
- 활동의 본질/속성 혹은 계약이나 법에서 요구되는 관계(프로토타입 제작 후 테스트 수행)
2) 임의적 의존관계 
- 모범적 실무관행 지식에 따라 설정되는 관계(분석 후 설계)
3) 외부 의존관계
- 프로젝트 활동과 비프로젝트 활동간의 관계(테스트는 하드웨어 납품되어야 수행 가능)
4) 내부 의존관계 
- 프로젝트 활동간 선/후행관계 의미(팀에서 기계조립 이후 테스트 수행)

### [프로젝트관리] PERT(Program Evaluation and Review Technique)

**[정의]**

- 프로젝트 활동의 기간을 불확실성(리스크)을 고려하여 확률적으로 산정하는 기법
- 낙관치(O)·최빈치(M)·비관치(P) 세 값으로 기대기간(ET)과 표준편차(σ)를 산출하는 삼점 산정 기법
- 기간 불확실성을 반영하여 현실적인 일정 계획 수립 지원

**[핵심/키워드]**

* PERT 핵심 공식
- 기대기간(ET, Expected Time) = (O + 4M + P) / 6
- 표준편차(σ) = (P - O) / 6
- 분산(V) = [(P - O) / 6]²

* 3가지 기간 추정치
- O(Optimistic, 낙관치) : 최이상적 조건에서 최소 소요기간
- M(Most Likely, 최빈치) : 가장 현실적·발생 가능성 높은 기간
- P(Pessimistic, 비관치) : 최악 상황에서 최대 소요기간

* 신뢰도 구간 (정규분포 가정)
- μ ± 1σ : 68.27% 확률로 완료
- μ ± 2σ : 95.45% 확률로 완료
- μ ± 3σ : 99.73% 확률로 완료

* PERT vs CPM 비교
- PERT : 확률적 기간 산정 (불확실성 반영), 연구·개발 프로젝트 적합
- CPM : 결정론적 기간 산정 (단일 추정치), 반복 건설 프로젝트 적합

* PERT/CPM 활용
- 임계경로 식별 → 일정 단축(Crashing/Fast Tracking) 대상 선정

### [프로젝트관리] CPM(Critical Path Method, 주공정경로법)

**[정의]**

- 프로젝트 전체 기간을 결정하는 가장 긴 경로인 Critical Path를 찾아내는 일정 분석 기법
- 프로젝트 내 활동 간의 논리적 순서(의존 관계)와 기간(Duration)을 바탕으로 최소 완료 기간을 산출하는 기법 

- 임계경로(Critical Path): 총 여유시간(Total Float)이 0인 활동들의 연속 경로
- 총 여유시간(Total Float): 활동이 전체 일정을 지연시키지 않고 지연될 수 있는 최대 시간 
- 조기착수(ES)/조기완료(EF), 후기착수(LS)/후기완료(LF) 개념을 통해 활동 일정을 계산 

* 수행 절차
1\. 활동 목록 작성 및 의존 관계 설정
2\. 활동 기간 산정
3\. 일정 네트워크도 작성 (PDM 활용)
4\. 순방향 계산(Forward Pass) – 조기착수/완료 시점 계산 
5\. 역방향 계산(Backward Pass) – 후기착수/완료 시점 계산 
6\. 총 여유시간(Total Float) 계산 및 임계경로 식별

**[핵심/키워드]**

* 주요 용어 
- 주공정(Critical Path): 프로젝트 시작에서 종료까지 가장 긴 시간이 걸리는 경로. 여유시간(Float)=0인 활동들의 집합

- ES(Early Start): 빠른 시작일
- EF(Early Finish): 빠른 종료일  
- LS(Late Start): 늦은 시작일 
- LF(Late Finish): 늦은 종료일 
- TF(Total Float): 총 여유 
- FF(Free Float): 자유여유 

- 전진계산(Forward Scheduling): 시작부터 순서대로 모든 활동의 ES, EF를 계산하는 방법(좌→우)
- 후진계산(Backword Scheduling): 종료부터 역순으로 모든 활동의 LF, LS를 계산하는 방법

### [프로젝트관리] CCM(Critical Chain Method, 주공정연쇄법)

**[정의]**

- 자원 제약 사항을 고려하여 여유시간을 줄여 통합된 버퍼로 책정하고 버퍼의 소진율을 모니터링 함으로써 전체 프로젝트 일정을 관리하는 방법
- 프로젝트 일정 개발 시, 자원제약과 불확실성을 함께 고려하여 일정을 최적화하는 자원기반 일정관리 기법

**[핵심/키워드]**

* 버퍼 종류: 프피자 
- Project Buffer(프로젝트 버퍼): 주공정의 마지막에 삽입되는 버퍼 
- Feeding Buffer(피딩 버퍼): 비임계경로(Non-critical Path)가 주공정연쇄에 합류하기 직전에 삽입
- 자원 버퍼(Resource Buffer): 중요 자원 투입 직전에 설정되는 경보(Alarm) 형태의 버퍼

### [프로젝트관리] 일정단축기법: Crashing (공정압축법)

**[정의]**

- 추가 자원을 투입하여 주공정(임계경로)상의 활동 기간을 단축하는 방법

- 접근방식: 비용(Cost)을 늘려 기간(Time)을 줄이는 방식

**[핵심/키워드]**

- 대표방법: 인력 추가 투입, 장비 증설, 초과 근무, 외주 발주

- 장점: 일정 단축 효과 확실 
- 단점: 비용 상승, 자원 과부하 발생 가능

### [프로젝트관리] 일정단축기법: Fast Tracking (공정중첩법)

**[정의]**

- 논리적으로 순차적(직렬)인 활동을 병렬로 겹쳐 수행하여 일정을 단축 하는 기법(병행수행)

- 접근방식: 작업 순서(Logic)를 변경하여 동시 수행(Parallel Work)

**[핵심/키워드]**

- 대표 방법: 병행 가능한 활동 구간 중첩 수행 

- 장점: 추가자원 투입 필요없음 
- 단점: 재작업 발생가능, 리스크 증가

### [프로젝트관리] 자원최적화기법: 자원평준화 (Resource Leveling)

**[정의]**

- 자원의 가용성을 기준으로 활동의 시작일·종료일을 조정하여 자원 수요와 공급의 균형을 확보하는 기법
- 자원 과부하(Over-allocation) 해소를 위해 활동 일정을 지연 가능하며, 임계경로(Critical Path) 변경 및 전체 일정 지연 가능
- Resource Smoothing(자원 평활화)과 구분: 평준화는 납기 변경 가능, 평활화는 납기 고정

**[핵심/키워드]**

* Resource Leveling(자원평준화)
- 목적 : 자원 가용성 기준으로 활동 일정 조정 → 과부하 해소
- 특징 : 임계경로 변경 가능, 전체 프로젝트 완료일 연장 가능
- 적용 : 자원이 특정 기간 과도하게 집중되거나 공유 자원이 부족할 때

* Resource Smoothing(자원평활화)와 비교
구분             | 자원평준화(Leveling)     | 자원평활화(Smoothing)
납기(Deadline) | 변경 가능                  | 고정 (변경 불가)
임계경로        | 변경될 수 있음            | 변경되지 않음
여유시간 활용  | 총여유 + 자유여유 모두  | 자유여유(Free Float) 범위 내
자원 제약       | 강한 제약 우선 반영      | 일정 제약 우선 반영

* 수행 순서
1\. 자원 과부하 구간 식별
2\. 여유(Float)가 있는 활동 일정 조정
3\. 필요 시 활동 기간·순서 변경
4\. 최종 일정 확정 및 기준선 재설정

* 관련 기법
- Heuristic 시행착오법, CPM/CCM과 연계

### [프로젝트관리] 자원최적화기법: 자원평활화 (Resource Smoothing)

**[정의]**

- 일정 제약을 기준으로 자원 사용량을 제한 내에서 조정 하는 기법

- 목적: 자원 사용을 일정하게 유지 

- 임계경로(Critical Path): 변경되지 않음(완료일 고정)

**[핵심/키워드]**

1) 초기상태(왼쪽 위) 
-  프로젝트 초기에 자원 사용량이 불균형한 상태 
- 일부 주(예: W7)에는 45시간 이상 과도한 자원 투입(Over-allocation) 발생 

2) Resource Leveling(오른쪽 위) 
- 자원 수요를 가용량(예: 45시간) 수준으로 균등하게 분배 
- 하지만 활동이 뒤로 밀리면서 전체 일정이 늘어남 

3) Resource Smoothing(아래쪽) 
- 일정이 연장된 상태에서 자원평활화(Smoothing)을 적용 
- 각 주의 자원 사용량을 더욱 균등하게 조정(예: 38~45시간 사이로 안정화)
- 납기(Deadline)는 그대로 유지하며, 여유시간(Float) 내 조정으로 효율성 향상

### [프로젝트관리] 획득가치관리(Earned Value Management, EVM) = 기성고 분석

**[정의]**

- 프로젝트의 범위, 일정, 자원 측정치를 모두 결합하여 프로젝트 성과 및 진척율을 평가하는 방법론 
- 프로젝트의 일정·원가·성과를 통합적으로 측정하기 위한 관리기법 

* 측정지표
- 계획가치(PV: Planned Value): 특정시점까지 완료해야하는 물리적 작업량 표현 
- 획득가치(EV: Earned Value): 특정시점까지 실제 수행한 작업에 할당되었던 예상금액 
- 실제원가(AC: Actual Cost): 특정시점까지 실제 수행한 작업에 할당되었던 예산금액 
- 완료시점예산(BAC: Budget At Completed): 수행할 작업에 대해 책정된 모든 예산의 합계

**[핵심/키워드]**

* 분석지표
- 원가차이(CV: Cost Variance): 원가성과에 대한 측정치, CV=EV-AC 
- 일정차이(SV: Schedule Variance): 일정성과에 대한 측정치, SV=EV-PV
- 원가성과지수(CPI: Cost Performance Index): 완료한작업의 원가효율성에 대한 측정치, CPI=EV/AC 
- 일정성과지수(SPI: Schedule Performance Index): 일정효율성에 대한 측정치, SPI=EV/PV 

* 예측지표 
- 잔여분 산정치(ETC: Estimate To Completion): 현재시점부터 프로젝트 완료시까지의 총 예상금액, ETC=BAC-EV 
- 완료시점 산정치(EAC: Estimate At Completion): 프로젝트 완료시까지의 총 예상금액, EAC=ETC+AC
- VAC(완료시점차이) : BAC - EAC( < 0 : 망함)
- 완료성과지수(TCPI: To-Complete Performance Index): 지정된 관리목표를 달성하기 위해 잔여 작업에서 반드시 달성해야만 하는 원가성과 측정치, TCPI=잔여작업/잔여잔금=BAC-EV/BAC-AC

### [프로젝트관리] 7QC 도구

**[정의]**

- 품질 문제를 시각화·분석·개선하기 위한 가장 기초적이고 핵심적인 통계적 품질관리 도구 7가지
- TQM·6시그마·ISO 품질개선 활동의 필수 구성요소로 데이터 분석(정량적) 중심
- 품질계획·관리·통제·문제해결 활동 전반에 활용

**[핵심/키워드]**

* 7QC 도구 (체히파산관원흐)

① 체크시트 (Check Sheet)
- 문제 발생 빈도·유형을 정량적으로 기록하는 표준화된 양식
- 데이터 수집·분류, 품질 문제 신속 파악

② 히스토그램 (Histogram)
- 측정 데이터를 막대그래프로 시각화 → 분포·편차 확인
- 공정 변동성·안정성 평가

③ 파레토도 (Pareto Chart)
- 80:20 법칙 기반, 소수 원인이 다수 문제 유발 시각화
- 개선 우선순위 설정

④ 산점도 (Scatter Diagram)
- 두 변수 간 관계를 점으로 표시 → 상관성 파악
- 예: 공정온도 ↔ 불량률

⑤ 관리도 (Control Chart)
- UCL/LCL 기준선으로 공정 안정성 모니터링
- Rule of Seven: 7개 연속 점이 평균 한쪽에 위치 시 이상

⑥ 원인결과도 (Fishbone/Ishikawa Diagram)
- 문제의 근본 원인을 6M(Man·Machine·Material·Method·Measurement·Environment)으로 분류

⑦ 흐름도 (Flow Chart)
- 프로세스 단계별 흐름 도식화 → 병목·비효율 식별

### [프로젝트관리] 7QC 도구: 관리도 (Control Chart)

**[정의]**

- 품질의 산포를 관리하기 위하여 하나의 중심선과 두 개의 관리 한계선(관리 상한선, 하한선)을 설정한 그래프 

- 일정 기간 동안의 측정값을 시간 순서대로 그래프화하여, 공정이 통계적 관리 상태에 있는지 판별(UCL/LCL 기준)

- 주요 목적: 공정의 안정성 모니터링 


- USL/LSL(최고/최저 관리한계): 프로세스의 허용 가능한 편차의 범위, 관리상태하에 있음(--점선표시)
- USL/LSL(최고/최저 규격한계): 성과나 품질에 대한 고객의 기대사항이나 계약적인 요구사항(ㅡ실선표시) 
- Out of Control: 프로세스가 관리하에 있지 않다는 의미, UCL/LCL 바깥쪽에 점이 있는 경우, Rule of Seven인 경우 

- Rule of Seven: Data Point가 연속적으로 평균 위에 혹은 아래에 7개 이상 위치하거나, 연속적으로 같은 기울기를 가지고 있는 경우 

- 이상원인(가피원인): 피할수 있는 원인(15~20%)→현장에서 시정조치 가능
- 불가피원인: 피할수 없는 원인(80~85%)→경영진 참여 필요

**[핵심/키워드]**

&lt; 측정 기준 &gt; 
1\. 중심선 (Center Line, CL)
- 공정 데이터의 평균값(μ)을 나타내는 기준선
- 공정의 기준 상태(Base Line) 역할 수행
2\. 상한 관리한계선 (Upper Control Limit, UCL)
- 공정 변동의 상한 통계적 한계
3\. 하한 관리한계선 (Lower Control Limit, LCL)
- 공정 변동의 하한 통계적 한계
4\. 상한 규격한계 (Upper Specification Limit, USL)
- 고객 또는 설계 요구사항이 정한 최대 허용값
- 공정 결과가 이 한계를 초과하면 불량(Out of Spec)으로 간주
5\. 하한 규격한계 (Lower Specification Limit, LSL)
- 고객 또는 설계 요구사항이 정한 최소 허용값
- 공정 결과가 이 한계보다 작으면 불량으로 판단
6\. 데이터 포인트 (Measured Data Points)
- 각 시점의 측정값을 점으로 표시
- 시간에 따른 공정의 변동 추세 시각화
- 이상치·패턴 분석을 통해 공정 이상을 조기 탐지

### [프로젝트관리] 신 7QC 도구

**[정의]**

- 단순 데이터 분석을 넘어 문제의 구조화·의사결정·계획 수립을 지원하는 관리·기획 중심의 품질 개선 도구 7가지
- 기획·분석·의사결정 중심(정성적), 기존 7QC 도구의 정량적 접근과 상호 보완
- 복잡한 문제의 원인-해결 경로 체계화 및 의사결정 합리화에 활용

**[핵심/키워드]**

* 신 7QC 도구 (친관계매우활PDPC)

① 친화도(Affinity Diagram) : 다수 아이디어를 주제별로 그룹화
② 상호관계도(Relation Diagram) : 원인-결과 간 복잡한 인과관계 도식화
③ 계통도(Tree Diagram) : 목표 달성 수단을 계층적으로 전개
④ 매트릭스도(Matrix Diagram) : 두 요소 집합 간 관계·강도 시각화
⑤ 우선순위지표(Matrix Data Analysis) : 매트릭스 데이터를 수치화·우선순위 결정
⑥ PDPC(Process Decision Program Chart) : 계획 실행 중 발생 가능한 문제·대응책 사전 도식화
⑦ 활동네트워크도(Activity Network Diagram) : 활동 순서·병행관계·시간 도식화 (PERT/CPM 유사)

* 기존 7QC vs 신 7QC 비교
구분  | 기존 7QC               | 신 7QC
특성  | 정량적(Quantitative)  | 정성적(Qualitative)
목적  | 공정 데이터 분석       | 문제 구조화·계획 수립
활용  | 실행·통제 단계        | 기획·계획·분석 단계
결과물| 그래프·통계표         | 도식·관계도

### [프로젝트관리] 신 7QC 도구: 활동네트워크도 (Activity Network Diagram)

**[정의]**

- 활동 간 순서·병행 관계·소요시간을 도식화하여 일정 최적화를 지원하는 신 7QC 도구
- PERT/CPM 기법과 유사하며 병목 구간·임계경로(Critical Path) 파악에 활용
- 프로젝트 계획·품질개선 활동의 실행 순서와 의존관계를 시각적으로 표현

**[핵심/키워드]**

* 활동네트워크도 구성요소
- 노드(Node) : 이벤트 또는 마일스톤을 나타내는 원
- 화살표(Arrow) : 활동 및 선후 관계를 표시
- 더미(Dummy Arrow) : 점선 화살표, 논리적 의존관계만 표시 (기간=0)

* 활동 표기 방식
- AOA(Activity On Arrow) : 화살표가 활동, 노드가 이벤트 → PERT 방식
- AON(Activity On Node) : 노드가 활동 → PDM(Precedence Diagram Method) 방식

* 임계경로(Critical Path) 도출
- 순방향 계산(Forward Pass) : ES·EF 산출
- 역방향 계산(Backward Pass) : LS·LF 산출
- Total Float = LS - ES = LF - EF
- Total Float = 0인 활동들의 경로 = 임계경로

* 활용
- 신 7QC의 하나로 품질개선 계획 수립 시 활용
- 공정 최적화·일정 단축 우선순위 결정

### [프로젝트관리] Responsibility Assignment Matrix(책임배정 매트릭스) – RACI Chart

**[정의]**

- 각 작업(Activity) 또는 산출물(Deliverable)에 대해 누가 어떤 역할을 담당하는지를 명확히 정의하는 도표 

- 행(Row): 프로젝트의 작업 또는 활동(Activity)
- 열(Column): 팀원·조직·역할(Role) 
- 교차 셀에 R,A,C,I를 표기하여 각 역할별 책임 정도를 시각화

**[핵심/키워드]**

&lt; RACI 구성요소 &gt;
- R (Responsible): 수행담당자 
– 실제로 작업을 수행하는 책임자
- A (Accountable): 총괄책임자 
– 최종 승인 및 의사결정 책임자 (한 활동당 1인 원칙) 
- C (Consulted): 자문자 
– 수행 중 의견이나 전문지식을 제공하는 사람
- I (Informed): 통보대상자 
– 진행상황이나 결과를 통보받는 이해관계자

&lt; RACI 작성원칙 &gt; 
- 한 업무에 대해 R/A는 필수 
- 한 업무에 A는 한 주체에게만 할당
- C/I는 생략가능 
- R/A, C/I 는 한 주체에게 동시에 할당 될 수 있음

### [프로젝트관리] Tuckman의 5단계 팀 개발 모델(Tuckman’s Team Development Model) = 터크만 사다리 모델

**[정의]**

- 브루스 터크만(Bruce Tuckman, 1965)이 제시한 팀 형성~고 성과 팀 성장까지의 심리적·행동적 변화 5단계 모델
- 프로젝트 관리자가 팀 발달 단계에 적합한 리더십(지시→코칭→참여→위임→해체)을 발휘하도록 지원
- 태동기~휴회기 단계 변화에 따라 팀 역학과 관리 전략이 달라짐 (태혼안성휴)

**[핵심/키워드]**

* 5단계 팀 개발 모델 (태혼안성휴 / 포스노페아)

① 태동기(Forming) - 형성
- 특징 : 역할·목표 탐색, 조심스럽고 예의바름, 방향 불명확
- 관리 : 목표·역할·의사소통 체계 명확히 제시 → 지시형(Directing) 리더십

② 혼돈기(Storming) - 격동
- 특징 : 의견 충돌·역할 갈등·리더십 도전, 감정적 반응
- 관리 : 갈등조정·팀 신뢰 형성·목표 재확인 → 코칭형(Coaching) 리더십

③ 안정기(Norming) - 규범화
- 특징 : 신뢰·협력관계 형성, 팀 규칙·역할 명확화
- 관리 : 협업 강화·동기부여 → 참여형(Supporting) 리더십

④ 성취기(Performing) - 성과
- 특징 : 자율적 고성과 달성, 문제해결 중심의 성숙한 협력
- 관리 : 성과 인정·지속적 개선 지원 → 위임형(Delegating) 리더십

⑤ 휴회기(Adjourning) - 해체
- 특징 : 프로젝트 종료·팀 해산, 일부 아쉬움·불안감
- 관리 : 성과 공유·교훈 도출·전이 지원 → 해체(Adjourning)

### [프로젝트관리] 동기부여이론

**[정의]**

- 조직 구성원의 의욕·참여·성과를 향상시키기 위한 관리 활동의 이론적 기반
- 무엇이(내용이론), 어떻게(과정이론), 어떤 결과가(강화이론) 행동을 동기부여하는지를 분석하는 이론 체계
- 프로젝트 관리자는 동기부여 이론을 이해하고 상황에 맞는 동기 전략 적용

**[핵심/키워드]**

* 동기부여 이론 3가지 유형

① 내용이론 (Content Theory) - 무엇이 동기 유발?
- 매슬로(Maslow) 욕구위계 : 생리→안전→소속→존경→자아실현 (5단계, 결핍욕구·성장욕구)
- 앨더퍼(Alderfer) ERG : 존재(E)·관계(R)·성장(G) 3단계, 다중 욕구 동시 작용·퇴행이론
- 허즈버그(Herzberg) 동기-위생이론 : 동기요인(성취·인정·책임→만족) vs 위생요인(급여·환경→불만방지)
- 맥클리랜드(McClelland) 3욕구 : 성취(Ach)·권력(Pow)·친화(Aff) 욕구
- 맥그리거(McGregor) X·Y이론 : X(수동적→통제관리) vs Y(자율적→참여관리)

② 과정이론 (Process Theory) - 어떻게 동기 유발?
- 브룸(Vroom) 기대이론 : 동기 = 기대(E) × 수단성(I) × 유의성(V)
- 애덤스(Adams) 형평성이론 : 투입/산출 비율의 공정성 인식
- 로크(Locke) 목표설정이론 : 구체적·도전적 목표 설정

③ 강화이론 (Reinforcement Theory) - 행동 강화·약화
- 스키너(Skinner) : 보상→행동 강화, 처벌→행동 약화
- 손다이크(Thorndike) 효과의 법칙 : 만족스러운 결과 → 행동 반복

### [프로젝트관리] 브룸(Vroom)의 기대이론 (Expectancy Theory)

**[정의]**

- 개인은 노력 → 성과 → 보상의 인과관계에 대한 기대 수준에 따라 동기부여됨을 정의한 이론 

- E×I×V = Motivation
노력의 결과가 보상으로 이어질 기대경로 명확화 필요

**[핵심/키워드]**

- 사람이 열심히 일하면 성과가 나고, 그 성과가 보상으로 이어질 것이라는 믿음이
강할수록 동기부여 수준이 높음
- 동기 = 기대(Expectancy) × 수단성(Instrumentality) × 유의성(Valence)

### [프로젝트관리] 손다이크(Thorndike)의 효과의 법칙 (Law of Effect)

**[정의]**

- 행동은 그 결과(보상 또는 처벌)에 따라 강화되거나 약화된다는 법칙 

- 행동은 결과에 의해 학습된다는 원리 제시 → 스키너 강화이론의 기초

**[핵심/키워드]**

- 만족스러운 결과를 가져오는 행동은 반복될 가능성 증가
- 불쾌한 결과를 초래하는 행동은 감소
- 고양이 퍼즐 상자 실험을 통해 보상 중심 학습 원리 입증

### [프로젝트관리] 스키너(Skinner)의 강화이론 (Reinforcement Theory)

**[정의]**

- 인간의 행동은 그 행동의 결과(보상·처벌)에 의해 강화되거나 약화된다고 보는 행동주의 이론 

- 행동조건이론(Operan t Conditioning) 기반 보상·처벌·소거에 따른 학습

**[핵심/키워드]**

- 행동 → 결과 → 다음 행동 강화 또는 약화
- 바람직한 행동은 보상으로 강화, 바람직하지 않은 행동은 처벌·소거로 약화

### [프로젝트관리] 리스크관리(위험관리)

**[정의]**

- 프로젝트 목표(범위, 일정, 비용, 품질 등)에 영향을 줄 수 있는 불확실성을 식별·분석·대응·감시하는 체계적인 과정을 정의한 지식영역 

- 부정적 리스크(위협)는 최소화하고, 긍정적 리스크(기회)는 극대화하여 프로젝트의 성공 가능성을 높이는 관리 활동

### [프로젝트관리] 확률-영향 매트릭스(Probability–Impact Matrix, P–I Matrix)

**[정의]**

- 리스크의 발생 가능성(Probability)과 프로젝트 목표에 미치는 영향도(Impact)를 결합하여 각 리스크의 상대적 우선순위를 시각적으로 표현하는 정성적 리스크 분석 도구
- 리스크를 위협(Threat)과 기회(Opportunity) 영역으로 구분하여 관리 방향 제시
- 정성적 리스크 분석의 핵심 도구로 대응 전략 수립의 기초 자료 제공

**[핵심/키워드]**

* P-I Matrix 구조
- X축 : 영향도(Impact) - 낮음(Low)/중간(Medium)/높음(High)
- Y축 : 발생가능성(Probability) - 낮음/중간/높음
- 셀 색상 : 빨강(High Risk) / 노랑(Medium Risk) / 초록(Low Risk)

* 위험(Threat) 대응 전략
- 회피(Avoid) : 리스크 원인 제거, 계획 변경
- 완화(Mitigate) : 발생가능성·영향도 감소
- 전가(Transfer) : 제3자(보험·외주)에게 이전
- 수용(Accept) : 능동적(예비비 설정) / 수동적(그냥 받아들임)

* 기회(Opportunity) 대응 전략
- 활용(Exploit) : 불확실성 제거, 기회 실현 확실화
- 강화(Enhance) : 발생가능성·긍정 영향도 증대
- 공유(Share) : 기회 포착 능력 있는 제3자와 협력
- 수용(Accept) : 기회 발생 시 이익 추구

* P-I Matrix 활용 절차
1\. 리스크 등록부에서 리스크 목록 확인
2\. 발생가능성·영향도 평가 (전문가 판단, 인터뷰)
3\. 매트릭스 배치 → 우선순위 결정
4\. 고위험 리스크 중심으로 대응계획 수립

### [프로젝트관리] 몬테카를로 시뮬레이션 (Monte Carlo Simulation)

**[정의]**

- 확률분포를 정의하고 무작위 난수를 이용해 반복 계산하는 확률적 분석 기법 

- 일정 리스크 분석에서는 기간 산정치와 일정 네트워크도, 원가 리스크 분석에서는 원가 산정치를 변수로 사용
- 반복 결과로 특정 납기·원가 목표 달성 확률을 계산

**[핵심/키워드]**

* 활용 도구 및 결과물 
- S-곡선(S-Curve), 확률분포 그래프, 달성확률 분석표


- 프로젝트의 비용·일정 목표에 대한 달성 확률을 시각화한 누적분포곡선(CDF, Cumulative Distribution Function)
- x축: 총비용 또는 총기간, y축: 누적확률(%)
- P50 지점: 50% 확률로 달성 가능한 목표(즉,“평균적 기대치”) 
- P80 지점: 80% 확률로 달성 가능한 목표(안전한-목표치, 예비비 반영 필요)
- 곡선 기울기: 완만할수록 불확실성(리스크)이 큼

### [프로젝트관리] 금전적 기대값 분석 (EMV, Expected Monetary Value Analysis)

**[정의]**

- 불확실한 사건의 발생 확률과 영향(손익)을 곱해 평균 기대값을 산출하는 통계 기법

- 확률×영향을 통한 기대 손익 계산 (리스크 중립 가정)
- 리스크 중립 가정(Risk-Neutral Assumption)을 기반으로 함
- 기회는 양수, 위협은 음수로 표현

**[핵심/키워드]**

* 활용 도구 및 결과물  
- EMV 계산표, 확률별 기대값 비교표

### [프로젝트관리] 리스크 대응전략: 위협 대응전략

**[정의]**

- 프로젝트 목표 달성을 위해 위협은 감소, 기회는 극대화하는 대응전략을 수립하는 프로세스
- 필요 시 예산·일정·계획에 반영

&lt;대응 계획의 요건&gt;
- 실현 가능하고, 리스크의 중요도에 적합하며, 비용 대비 효과 필요
- 모든 이해관계자의 합의와 명확한 책임자 지정 필요

**[핵심/키워드]**

1) 회피(Avoid): 위협을 제거하거나 위협으로부터 보호하기 위해 취하는 대응전략
- 일정연장, 전략변경, 범위축소, 프로젝트 중단  

2) 전가(Transfer): 위협의 영향을 제3자에게 이전하는 대응전략
- 보험활용, 이행보증, 각종 보증 및 보장, 외부 계약

3) 완화(Mitigate): 리스크 발생가능성 혹은 영향력을 감소시키는 대응전략
- 덜 복잡한 프로세스 채택, 더 많은 테스트 수행, 프로토타입 개발 

4) 수용(Accept): 리스크를 파악했으나 리스크가 발생하기 전에는 어떤 조치도 취하지 않는 대응전략 
- 능동적 수용전략, 수동적 수용전략 

5) 에스컬레이션: 위협이 프로젝트 범위를 벗어나 제안된 대응책이 PM의 권한을 넘어설 수 있다는 데 동의하는 경우 사용

### [프로젝트관리] 리스크 대응전략: 기회 대응전략

**[정의]**

- 프로젝트 목표 달성을 위해 위협은 감소, 기회는 극대화하는 대응전략을 수립하는 프로세스
- 필요 시 예산·일정·계획에 반영

&lt;대응 계획의 요건&gt;
- 실현 가능하고, 리스크의 중요도에 적합하며, 비용 대비 효과 필요
- 모든 이해관계자의 합의와 명확한 책임자 지정 필요

**[핵심/키워드]**

1) 활용(Exploit): 기회가 확실히 실현되도록 불확실성을 제거하는 방법

2) 강화/증대(Enhance): 기회의 확률 및 긍정적 영향을 증가시키기 위해 사용하는 전략

3) 공유/분담(Share): 프로젝트에 유리한 기회를 가장 잘 포착할 수 있는 제 3자에게 기회 소유권을 할당하는 것
- 협력사/팀/특수 목적회사/합작회사와 협력관계 구축 

4) 수용(Accept): 기회가 발생한다면 이익을 추구하지만, 발생하기 전에는 적극적으로 추구하지 않는 것

5) 에스컬레이션: 위협이 프로젝트 범위를 벗어나 제안된 대응책이 PM의 권한을 넘어설 수 있다는 데 동의하는 경우 사용

**[정의]**

[두음] 고원시
1) 고정가 계약: 공급받기로 한 제품, 서비스 및 결과물에 대한 고정가 총액을 설정하는 계약유형

- FP-EPA(가격조정 조건부 고정가계약): 고정가+물가반영

### [프로젝트관리] 이해관계자 분석 모델: 현저성 모델 (Salience Model)

**[정의]**

- 이해관계자의 중요도(Salience)를 권력(Power)·정당성(Legitimacy)·긴급성(Urgency) 3요소로 평가하여 우선순위를 결정하는 이해관계자 분석 모델
- 3요소 조합에 따라 이해관계자를 7가지 유형으로 분류하고 집중 관리 대상 식별
- Mitchell·Agle·Wood(1997)가 제안

**[핵심/키워드]**

* 3대 요소
- 권력(Power) : 영향력을 행사할 수 있는 능력
- 정당성(Legitimacy) : 요구가 조직적으로 타당한가 여부
- 긴급성(Urgency) : 즉시 조치가 필요한 요구 수준

* 7가지 이해관계자 유형
① 무시된(Dormant) - 권력만 : 권력 보유, 현재 영향 미미
② 잠재적(Discretionary) - 정당성만 : 요구는 타당하나 권력 없음
③ 요청적(Demanding) - 긴급성만 : 즉각 관심 요구, 영향력 약함
④ 지배적(Dominant) - 권력+정당성 : 공식 권한, 주요 내부 관계자
⑤ 위험한(Dangerous) - 권력+긴급성 : 압박 강함, 갈등 가능성 높음
⑥ 의존적(Dependent) - 정당성+긴급성 : 요구는 타당하나 권력 부족
⑦ 결정적(Definitive) - 권력+정당성+긴급성 : 세 속성 모두 보유, 최우선 관리

* 활용 전략
- 결정적(Definitive) → 최우선 집중 관리
- 지배적(Dominant) → 정기 보고·참여 유도
- 잠재적(Discretionary) → 정보 공유·관계 유지
- 요청적(Demanding) → 불만 관리, 리스크 모니터링

* 비교 : Power/Interest Grid(2×2) vs Salience Model(3요소)

### [UML/디패] 객체지향 (Object Oriented)

**[정의]**

- 현실세계의 개념들을 객체 단위로 구성하고, 객체들이 메시지를 통해 상호작용함으로써 전체 시스템이 운영되는 소프트웨어 개발 패러다임
- 데이터(속성)와 행위(메서드)를 하나의 객체로 묶어 모듈화하고, 재사용성·유지보수성·확장성을 향상시키는 개발 방법론
- 절차적 프로그래밍(Procedural)의 한계를 극복하기 위해 등장

**[핵심/키워드]**

* 주요 요소
- 객체(Object) : 현실 개념을 소프트웨어로 표현하는 기본 단위 (속성+행위)
- 클래스(Class) : 객체를 생성하기 위한 설계도·템플릿
- 메시지(Message) : 객체 간 통신 수단 (메서드 호출)
- 인스턴스(Instance) : 클래스로부터 생성된 실제 객체

* 4대 특징 (캡추다상)
- 캡슐화(Encapsulation) : 데이터+메서드 묶음, 내부 구현 은닉
- 추상화(Abstraction) : 공통 속성·행위만 추출하여 표현
- 다형성(Polymorphism) : 같은 이름, 다른 동작 (오버로딩·오버라이딩)
- 상속성(Inheritance) : 부모 클래스 속성·메서드 재사용

* SOLID 5원칙
- SRP(단일책임), OCP(개방폐쇄), LSP(리스코프치환), ISP(인터페이스분리), DIP(의존역전)

* 언어 예시 : Java, C++, Python, C#

### [UML/디패] 다형성 (Polymorphism)

**[정의]**

여러 클래스에 같은 이름의 함수가 존재하지만 동작은 다르게 수행하는 특징

* 같은 이름, 다른 동작

**[핵심/키워드]**

[ 구현 ]
1) 오버로딩 (Overloading) 
- 동일 클래스, 같은 이름의 오퍼레이션, 이름은 같고 파라미터의 개수와 형식은 다름

2) 오버라이딩 (Overriding) 
- 부모의 상속받은 오퍼레이션 자식 클래스가 같은 이름으로 재정의, 파라미터의 개수와 타입 동일

### [UML/디패] 정보은닉 (Information Hiding)

**[정의]**

- 메시지만으로 객체와 상호작용하도록 정보를 숨겨서 접근을 제한하는 특징
- 객체의 속성과 구현 세부사항을 외부에 감추고, Getter/Setter 같은 공개 메서드를 통해서만 접근하게 함으로써 데이터 보호와 무결성을 유지하는 객체지향의 특징

**[핵심/키워드]**

- 접근 제한자를 사용하여 외부 접근 차단
- 캡슐화를 통해 정보은닉 가능

### [UML/디패] 객체지향 설계 5대 원칙: SOLID

**[정의]**

- 객체지향방법론이 지향하는 재사용과 유지보수 극대화를 달성하기 위한 클래스 단위에서의 설계 원칙
- 로버트 마틴(Robert C. Martin)이 정립한 OOP 설계의 핵심 가이드라인
- 변경에 유연하고 결합도(Coupling)는 낮추며 응집도(Cohesion)는 높이는 것을 목표

**[핵심/키워드]**

* SOLID 5원칙

① SRP (Single Responsibility Principle, 단일책임 원칙)
- 한 클래스는 오직 하나의 책임만 가져야 함
- 역할 분리하여 응집도 향상, 유지보수 용이

② OCP (Open-Closed Principle, 개방폐쇄 원칙)
- 모듈은 확장에는 열려(Open) 있고 변경에는 닫혀(Closed) 있어야 함
- Open : 클래스 수직 관계(Is-a)에서는 열려 있어야 함
- Closed : 클래스 수평 관계(has-a)에서는 영향을 받지 않아야 함
- 구현 방법 :변하는(확장되는)것과 변하지 않는 것을 엄격히 구분한 후 이 두 모듈이 만나는 지점에 인터페이스를 정의

③ LSP (Liskov Substitution Principle, 리스코프 치환 원칙)
- 자식 클래스는 부모 클래스를 대체할 수 있어야 함
- 다형성을 보장하며,부모 클래스를 사용하는 곳에서 자식 클래스를 문제없이 사용할 수 있도록 설계 
- 상속 관계에서 부모 객체를 자식 객체로 대체 가능

④ ISP (Interface Segregation Principle, 인터페이스 분리 원칙)
- 인터페이스는 사용하지 않는 기능을 강요하지 않아야 함
- 하나의 큰 인터페이스보다 작은 여러 개의 인터페이스로 분리

⑤ DIP (Dependency Inversion Principle, 의존관계 역전 원칙)
- 고수준 모듈이 저수준 모듈에 직접 의존하지 말고 추상화에 의존해야 함
- 구체 클래스가 아닌 인터페이스·추상클래스에 의존

### [UML/디패] Builder Pattern

**[정의]**

- 복잡한 객체를 생성하는 방법과 표현하는 방법을 정의하는 클래스를 별도로 분리해 서로 다른
표현이라도 이를 생성할 수 있는 동일한 구축 공정을 제공
- 제품은 다르지만 같은 공정으로 생성 할 수 있게 하는 디자인 패턴

* 생성과 표현 분리, 독립적인 기능설계
* 제품은 다르지만 같은 공정으로 생성

**[핵심/키워드]**

[ 구성요소 ]
Builder : 인스턴스 생성 위한 인터페이스 결정 (생성부)
Concrete Builder : 인터페이스 구현 (표현) 
Director : Builder 인터페이스를 사용, 원하는 부품을 조립후, 인스턴스를 생산 (빌더) 
Product : 최종적으로 생산된 제품

### [UML/디패] Prototype Pattern

**[정의]**

원형이 되는(Prototypical) 인스턴스를 사용하여 생성할 객체의 종류를 명시하고 이렇게 만든 견본(클론)을 복사해서 새로운 객체를 생성하는 패턴

**[핵심/키워드]**

[ 구성요소 ]
Prototype : 자신을 복제하는데 필요한 인터페이스를 정의
Concrete Prototype : 자신을 복제하는 오퍼레이션을 구현
Client : Prototype에 복제를 요청함으로써 새로운 객체 생성

### [UML/디패] Adapter Pattern

**[정의]**

- 클래스의 재사용성을 높이기 위해 요구되는 특정 인터페이스로 변환·적용하여 클래스 간 호환성을 확보하는 구조 패턴
- 호환되지 않는 기존 클래스의 인터페이스를 클라이언트가 요구하는 인터페이스로 변환하여 재사용성을 높이는 구조 패턴
- 기존 코드를 수정하지 않고 재사용 가능하게 하는 래퍼(Wrapper) 패턴

**[핵심/키워드]**

* 구성요소
- Client : 어떤 행위가 필요한 객체 (예:12V가 필요한 노트북)
- Target : Client에게 필요한 메서드를 제공하는 인터페이스(예:12V 전원)
- Adaptee : 이미 존재하는 기능을 가진 클래스 (예:100V 전원. 재사용 대상)
- Adapter : Adaptee의 메서드를 활용하여 Target 인터페이스 구현(예:100V를 12V로 바꿔주는 어댑터)

* 2가지 구현 방식
① 클래스 Adapter (상속 기반)
- Adapter가 Adaptee 클래스를 상속(Inheritance)하여 구현
- 단일 상속만 지원. 다중상속 제약

② 객체 Adapter (위임 기반)
- Adaptee 객체를 내부에 포함(Composition)하여 위임
- 상속보다 유연, 런타임 교체 가능
- 권장 방식

### [UML/디패] Bridge Pattern

**[정의]**

- 기능의 클래스 계층과 구현의 클래스 계층을 분리하여 각각 독립적으로 확장 가능하도록 하는 구조 패턴
- 상속(Inheritance)을 통한 기능 확장과 위임(Delegation)을 통한 구현 분리를 동시에 지원
- 기능과 구현의 결합도를 낮춰 두 계층 모두 독립적으로 변경·확장 가능

**[핵심/키워드]**

* 구성요소
- Abstraction : 기능 클래스 계층의 최상위 클래스, Implementor 참조
- Refined Abstraction: Abstraction을 확장(상속)하여 기능 추가
- Implementor : 구현 클래스 계층의 최상위 인터페이스
- Concrete Implementor : Implementor 인터페이스를 실제로 구현

* Bridge Pattern 적용 시점
- 기능과 구현이 독립적으로 확장되어야 할 때
- 런타임에 구현을 바꾸고 싶을 때
- 상속 기반 설계로 클래스 수가 급격히 증가하는 것을 막고 싶을 때
